# 首页 — 后端接口设计

------

## 一、接口总览

```plain
首页各区块独立加载，前端并行请求，互不阻塞
共 7 个接口
```

| #    | Method | URI                         | 对应区块               |
| ---- | ------ | --------------------------- | ---------------------- |
| 1    | `GET`  | `/api/home/init`            | 首页聚合初始化（可选） |
| 2    | `GET`  | `/api/shop-type/list`       | 分类图标导航栏         |
| 3    | `GET`  | `/api/seckill/list`         | 限时秒杀专区           |
| 4    | `GET`  | `/api/shop/nearby`          | 附近商户               |
| 5    | `GET`  | `/api/shop/recommend`       | 猜你喜欢（AI 推荐）    |
| 6    | `GET`  | `/api/message/unread-count` | 通知铃铛未读数         |
| 7    | `GET`  | `/api/banner/list`          | 轮播图 Banner          |

------

## 二、接口详细设计

------

### 接口 1：首页聚合初始化（可选）

```plain
GET /api/home/init

说明：一次请求聚合多个区块数据，减少弱网下的请求次数
      前端也可选择拆分为独立接口并行调用
```

#### 请求参数（Query）

| 参数        | 类型   | 必填 | 说明                               |
| ----------- | ------ | ---- | ---------------------------------- |
| `longitude` | Double | 否   | 用户经度                           |
| `latitude`  | Double | 否   | 用户纬度                           |
| `cityCode`  | String | 否   | 城市编码（定位失败时前端传缓存值） |

#### 请求示例

```http
GET /api/home/init?longitude=116.481&latitude=39.997

Authorization: Bearer <token>     ← 可选
```

#### 成功响应

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "city": "滁州",
    "unreadCount": 3,
    "banners": [ "..." ],
    "shopTypes": [ "..." ],
    "seckillVouchers": [ "..." ],
    "nearbyShops": [ "..." ],
    "recommendShops": [ "..." ]
  }
}
实现策略：
  服务端内部并行调用各子服务（CompletableFuture）
  任一子模块超时不阻塞整体，降级返回空数组
```

#### 响应字段说明

| 字段           | 类型      | 说明                            |
| -------------- | --------- | ------------------------------- |
| `records`      | Array     | 商户列表                        |
| `hasMore`      | Boolean   | 是否有更多数据                  |
| `lastDistance` | Double    | 本页最后一条的距离（下页游标）  |

**records 数组元素字段：**

| 字段           | 类型     | 说明                            |
| -------------- | -------- | ------------------------------- |
| `id`           | Long     | 商户 ID                         |
| `name`         | String   | 商户名称                        |
| `image`        | String   | 商户主图（仅 1 张，列表缩略图） |
| `typeName`     | String   | 类型名称                        |
| `typeIcon`     | String   | 类型图标                        |
| `score`        | Double   | 综合评分                        |
| `avgPrice`     | Integer  | 人均价格（元）                  |
| `distance`     | Integer  | 距离（米）                      |
| `distanceText` | String   | 距离文案（500m / 1.2km）        |
| `aiSummary`    | String   | AI 口碑一句话摘要               |
| `tags`         | String[] | 标签（最多 2 个）               |

#### 设计说明

```plain
Redis GEO 天然限制：
  GEORADIUS / GEOSEARCH 只能返回 COUNT 限制的结果集
  不提供"满足半径条件的总数"

滚动加载优势：
  1. 无需计算 total，避免全量查询或 DB 兜底
  2. 用 lastDistance 做游标，天然匹配 GEO 的距离排序
  3. 商户密集区也能高效分页，不会返回上千条数据
```

#### 异常响应

| code | msg                   | 触发条件       |
| ---- | --------------------- | -------------- |
| 400  | 经纬度不能为空        | 未传坐标       |
| 400  | 搜索半径不能超过 10km | radius > 10000 |

## 三、核心实现逻辑

### 3.1 首页加载时序

```plain
客户端启动首页
│
├── 并行请求（不依赖登录态的接口先发）
│   │
│   ├── ① GET /api/banner/list
│   ├── ② GET /api/shop-type/list
│   ├── ③ GET /api/seckill/list
│   ├── ④ GET /api/shop/nearby
│   │
│   └── 依赖 Token 的接口
│       ├── ⑤ GET /api/message/unread-count
│       └── ⑥ GET /api/shop/recommend
│
└── 用户主动操作
    └── ⑦ 点击 [换一批] → GET /api/shop/recommend?excludeIds=...
```

### 3.2 秒杀列表查询

```sql
-- 查询当前进行中 + 即将开始的秒杀券
SELECT v.id AS voucher_id,
       v.shop_id,
       s.name AS shop_name,
       v.title,
       v.pay_value,
       v.actual_value,
       sv.stock,
       sv.total_stock,
       sv.begin_time,
       sv.end_time
  FROM tb_voucher v
  JOIN tb_seckill_voucher sv ON sv.voucher_id = v.id
  JOIN tb_shop s             ON s.id = v.shop_id
 WHERE v.type    = 1                          -- 秒杀券
   AND v.status  = 0                          -- 上架中
   AND v.deleted = 0
   AND sv.end_time > NOW()                    -- 未结束
   AND sv.deleted  = 0
 ORDER BY
       CASE
         WHEN sv.begin_time <= NOW() AND sv.end_time > NOW()
           THEN 0                             -- 进行中排前面
         ELSE 1                               -- 未开始排后面
       END,
       sv.begin_time ASC
 LIMIT #{pageSize};
```

### 3.3 附近商户查询（Redis GEO）

```plain
Service 流程：
│
├── 1. Redis GEORADIUS 查询
│     GEORADIUS shop:geo:all
│       #{longitude} #{latitude}
│       #{radius} m
│       WITHDIST ASC
│       COUNT #{pageSize}
│     → 得到 [(shopId, distance), ...]
│
├── 2. 批量查询商户详情
│     优先从 Redis Hash 读取
│     shop:cache:{shopId} → 商户基本信息
│     未命中则查库并回写缓存
│
├── 3. 补充关联数据
│     ├── 类型名称 → shop:type:{typeId}
│     ├── 标签     → shop:tags:{shopId}
│     └── AI 摘要  → shop:ai:summary:{shopId}
│
├── 4. 格式化距离文案
│     ├── < 1000m → "500m"
│     └── ≥ 1000m → "1.2km"
│
└── 5. 组装 VO 返回
```

### 3.4 AI 推荐策略

```plain
GET /api/shop/recommend 核心逻辑
│
├── A. 已登录用户（个性化推荐）
│   │
│   ├── 1. 用户画像构建
│   │   ├── 浏览历史 (tb_browse_history)    → 类型偏好
│   │   ├── 收藏记录 (tb_favorite)          → 价位偏好
│   │   ├── 消费记录 (tb_voucher_order)     → 品类偏好
│   │   └── 搜索历史 (tb_search_history)    → 兴趣关键词
│   │
│   ├── 2. 候选集生成
│   │   ├── 召回策略 1: 偏好类型下高分商户
│   │   ├── 召回策略 2: 相似用户喜欢的商户（协同过滤）
│   │   ├── 召回策略 3: 附近热门商户（有坐标时）
│   │   └── 召回策略 4: 最新入驻高分商户（新鲜度）
│   │
│   ├── 3. 排序打分
│   │   score = 0.30 × 偏好匹配度
│   │        + 0.25 × 商户评分归一化
│   │        + 0.20 × 距离归一化
│   │        + 0.15 × 热度归一化
│   │        + 0.10 × 新鲜度
│   │
│   ├── 4. 去重过滤
│   │   ├── 排除 excludeIds（换一批已展示的）
│   │   ├── 排除近期浏览过的（避免重复推荐）
│   │   └── 排除已收藏的（已知晓的商户）
│   │
│   └── 5. 生成推荐理由
│       ├── "根据您常吃川菜推荐"
│       ├── "您收藏的同类商户"
│       ├── "附近新开高分店"
│       └── "和您口味相似的人都在去"
│
└── B. 未登录用户（热门推荐）
    │
    ├── 按城市热度排序
    ├── 有坐标 → 附近高分商户
    ├── 无坐标 → 全城热门 Top N
    └── recommendReason = "本地热门"
```

### 3.5 换一批实现

```plain
前端调用：
GET /api/shop/recommend?pageSize=6&excludeIds=3001,3002,3003,3004,3005,3006

后端逻辑：
├── 从候选集中排除 excludeIds
├── 取下一批 pageSize 条
├── 候选集耗尽时循环复用（重新推荐但随机打散顺序）
└── 返回全新的推荐列表
```

### 3.6 倒计时同步方案

```plain
客户端倒计时校准：

1. 响应中返回 serverTime + remainSeconds
2. 前端计算本地时间与服务器时间的 offset
   offset = serverTime - localTime
3. 倒计时基于 remainSeconds 递减
4. 每 60 秒静默校准一次（轮询或 WebSocket）
5. 库存通过 WebSocket 或轮询刷新（10 秒级别）
```

## 五、秒杀库存防超卖

```plain
下单链路（非本页接口，但首页展示依赖其库存数据）

1. Redis 预扣库存
   DECR seckill:stock:{voucherId}
   → 返回值 < 0 → 库存不足 → INCR 回补 → 返回"已抢光"

2. Lua 脚本保证原子性
   ┌─────────────────────────────────────┐
   │ -- 判断库存 + 一人一单              │
   │ local stock = redis.call(           │
   │   'GET', KEYS[1])                   │
   │ if tonumber(stock) <= 0 then        │
   │   return -1  -- 库存不足            │
   │ end                                 │
   │ local exists = redis.call(          │
   │   'SISMEMBER', KEYS[2], ARGV[1])   │
   │ if exists == 1 then                 │
   │   return -2  -- 重复下单            │
   │ end                                 │
   │ redis.call('DECR', KEYS[1])         │
   │ redis.call('SADD', KEYS[2], ARGV[1])│
   │ return 0  -- 成功                   │
   └─────────────────────────────────────┘

3. 成功后发 MQ 异步创建订单 + 扣 DB 库存
```
