## 二、全局业务类型编码

> 所有涉及 `biz_type` / `target_type` 的表统一使用以下编码，避免跨表混淆：

| 编码 | 含义 | 常量名 | 涉及表 |
|------|------|--------|--------|
| 1 | 商户 | `BIZ_TYPE_SHOP` | tb_tag_relation, tb_favorite, tb_browse_history, tb_report |
| 2 | 笔记 | `BIZ_TYPE_BLOG` | tb_tag_relation, tb_favorite, tb_browse_history, tb_report |
| 3 | 评论 | `BIZ_TYPE_COMMENT` | tb_report |
| 4 | 用户 | `BIZ_TYPE_USER` | tb_report |

> DDL 变更：`tb_tag_relation.biz_type` 注释改为 `'业务类型(1商户,2笔记)'`，`tb_report.target_type` 注释改为 `'目标类型(1商户,2笔记,3评论,4用户)'`。详见文末 DDL 变更清单。

## 三、接口总览

| # | 方法 | 路径 | 说明 | 登录 |
|---|------|------|------|------|
| 1 | GET | `/api/blog/{id}` | 笔记详情 | 可选 |
| 2 | GET | `/api/blog/{id}/ai-summary` | AI 摘要 | 可选 |
| 3 | GET | `/api/blog/{id}/comments` | 评论列表（顶级分页 + 子评论预加载前3条） | 可选 |
| 4 | POST | `/api/blog/{id}/comments` | 发表评论 | **必须** |
| 5 | PUT | `/api/blog/{id}/favorite` | 收藏 / 取消收藏 | **必须** |
| 6 | PUT | `/api/follow/{userId}` | 关注 / 取消关注 | **必须** |
| 7 | PUT | `/api/blog/comments/{commentId}/like` | 评论点赞 / 取消点赞（**新增**） | **必须** |
| 8 | GET | `/api/blog/comments/{commentId}/replies` | 展开子评论（**新增**） | 可选 |
| — | PUT | `/api/blog/{id}/like` | 笔记点赞（Page5 已设计） | **必须** |

### 错误响应

| code | 场景 |
|------|------|
| 404 | 笔记不存在 / 已删除 |
| 403 | 笔记已屏蔽（status=2） |

### 业务流程

```
客户端 GET /api/blog/10001
          │
          ▼
┌──────────────────────────┐
│ ① 查询笔记基本信息         │ ← tb_blog (status=1, deleted=0)
│   不存在 → 404            │
│   status=2 → 403          │
└──────────┬───────────────┘
           │
     ┌─────┼─────────┬──────────────┐
     ▼     ▼         ▼              ▼
┌────────┐┌────────┐┌────────────┐┌───────────┐
│ tb_user ││tb_shop ││tb_tag_     ││ 交互状态    │

                              ┌────────┼────────┐

                         │isLiked ││isFav  ││isFoll  │
                         │点赞?   ││收藏?  ││关注?   │
                         └────────┘└───────┘└────────┘
                              │
                              ▼
                    ┌──────────────────┐
                    │ ② 浏览历史        │ ← 异步写入
                    │ 已登录 → 写入     │
                    │ 未登录 → 跳过     │
                    └──────────────────┘
```

### 核心 SQL

```sql
-- 主查询: 笔记 + 作者 + 商户 + 收藏数
SELECT
    b.id, b.title, b.content, b.images,
    b.like_count, b.comment_count, b.favorite_count,
    b.create_time,
    -- 作者
    u.id AS user_id, u.nick_name, u.avatar,
    -- 商户
    s.id AS shop_id, s.name AS shop_name,
    s.address AS shop_address, s.avg_price
FROM tb_blog b
JOIN tb_user u ON u.id = b.user_id AND u.deleted = 0
LEFT JOIN tb_shop s ON s.id = b.shop_id AND s.deleted = 0
WHERE b.id = #{blogId}
  AND b.status = 1
  AND b.deleted = 0;

-- 标签列表（biz_type=2 表示笔记，全局统一编码）
SELECT t.id, t.name
FROM tb_tag_relation tr
JOIN tb_tag t ON t.id = tr.tag_id AND t.deleted = 0
WHERE tr.biz_type = 2
  AND tr.biz_id = #{blogId};

-- 以下仅已登录时执行 ↓↓↓

-- 点赞状态
SELECT 1 FROM tb_blog_likes
WHERE blog_id = #{blogId} AND user_id = #{currentUserId};

-- 收藏状态（biz_type=2 表示笔记）
SELECT 1 FROM tb_favorite
WHERE biz_type = 2 AND biz_id = #{blogId}
  AND user_id = #{currentUserId} AND deleted = 0;

-- 关注状态
SELECT 1 FROM tb_follow
WHERE user_id = #{currentUserId} AND follow_user_id = #{authorId}
  AND deleted = 0;

-- 浏览历史（异步写入，仅已登录）
INSERT INTO tb_browse_history (user_id, biz_type, biz_id, view_time, view_count)
VALUES (#{userId}, 2, #{blogId}, NOW(), 1)
ON DUPLICATE KEY UPDATE
    view_time = NOW(),
    view_count = view_count + 1;
```

## 五、接口 2 — AI 摘要

独立接口，**懒加载**，首次请求时生成并缓存。

### 基本信息

```
GET /api/blog/{id}/ai-summary
Authorization: Bearer <token>          # 可选
```

### 成功响应

```json
{
  "code": 200,
  "msg": "ok",
  "data": {
    "summary": "推荐川味小馆的水煮鱼和毛血旺，人均85元，环境好，适合朋友聚餐。建议提前预约。",
    "cached": true,
    "fallback": false
  }
}
```

| 字段 | 说明 |
|------|------|
| `summary` | 摘要文本 |
| `cached` | 是否来自缓存 |
| `fallback` | 是否为降级内容（LLM 不可用时截取正文前100字） |

### 错误响应

| code | 场景 |
|------|------|
| 404 | 笔记不存在 |

### 业务流程（含降级）

```
客户端 GET /api/blog/10001/ai-summary
          │
          ▼
┌──────────────────────────┐
│ 查 Redis 缓存             │
│ key: blog:summary:10001  │
└──────────┬───────────────┘
           │
      ┌────┴────┐
      │ 命中?    │
      └────┬────┘
     是    │    否
      ▼    │    ▼
 ┌────────┐│ ┌────────────────────────┐
 │ 直接返回 ││ │ 查 tb_blog 获取          │
 │ cached ││ │ title + content         │
 │ = true ││ └───────────┬────────────┘
 └────────┘│             ▼
           │  ┌────────────────────────┐
           │  │ 调用 LLM API (超时5s)    │
           │  │ (DeepSeek / GPT / 通义) │
           │  │                        │
           │  │ Prompt:                │
           │  │ "请用50字内总结这篇探店   │
           │  │  笔记的核心推荐要点"     │
           │  └───────────┬────────────┘
           │         ┌────┴────┐
           │         │ 成功?    │
           │         └────┬────┘
           │        是    │    否（超时/异常/熔断）
           │         ▼    │    ▼
           │   ┌──────────┐  ┌──────────────────┐
           │   │ 写 Redis  │  │ 降级: 截取正文     │
           │   │ TTL=7天   │  │ 前100字作为摘要    │
           │   │ fallback  │  │ fallback=true     │
           │   │ = false   │  │ 不缓存(下次重试)   │
           │   └──────────┘  └──────────────────┘
           │         │              │
           └─────────┴──────────────┘
                     │
                     ▼
                返回摘要结果
```

**降级策略**：
- LLM 调用超时设为 **5 秒**
- 连续失败 **3 次**后触发熔断，30 秒内直接走降级
- 降级内容（`fallback=true`）**不写入缓存**，下次请求重试 LLM

```
└─────────────────────────────────┘
```

### 核心 SQL

```xml
<!-- 
  第1步: 查询顶级评论（分页）
  使用动态 SQL 代替 CASE WHEN，确保 MySQL 能走索引排序
-->
<select id="selectTopComments" resultType="...">
    SELECT
        c.id, c.content, c.like_count, c.create_time,
        u.id AS user_id, u.nick_name, u.avatar,
        (SELECT COUNT(*) FROM tb_blog_comments sub
         WHERE sub.root_comment_id = c.id
           AND sub.deleted = 0 AND sub.status = 0
        ) AS reply_count
    FROM tb_blog_comments c
    JOIN tb_user u ON u.id = c.user_id AND u.deleted = 0
    WHERE c.blog_id = #{blogId}
      AND c.root_comment_id = 0
      AND c.status = 0
      AND c.deleted = 0
    <choose>
        <when test="sortBy == 'hot'">
            ORDER BY c.like_count DESC, c.create_time DESC
        </when>
        <otherwise>
            ORDER BY c.create_time DESC
        </otherwise>
    </choose>
    LIMIT #{offset}, #{size}
</select>
```

```sql
-- 第2步: 批量查询每条顶级评论的热门前3条子回复（窗口函数截断）
```

## 七、接口 4 — 发表评论

### 基本信息

```
POST /api/blog/{id}/comments
Content-Type: application/json
Authorization: Bearer <token>          # 必须
```

### 路径参数

| 参数 | 类型 | 说明 |
|------|------|------|
| `id` | Long | 笔记ID |

### 请求体

```json
{
  "content": "看了就想去！已收藏",
  "rootCommentId": 0,
  "replyCommentId": 0
}
```

### 参数说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `content` | String | **是** | 评论内容，1~2048字符 |
| `rootCommentId` | Long | 否 | 顶级评论ID（0=发表顶级评论） |
| `replyCommentId` | Long | 否 | 回复目标评论ID（0=非楼中楼回复） |

### 场景映射

```
场景1: 直接评论笔记 (顶级评论)
  rootCommentId = 0,  replyCommentId = 0

场景2: 回复某条顶级评论
  rootCommentId = 50001,  replyCommentId = 50001

场景3: 回复楼中楼的某条回复
  rootCommentId = 50001,  replyCommentId = 50010
```

### 成功响应

```json
{
  "code": 200,
  "msg": "评论成功",
  "data": {
    "commentId": 50088,
    "createTime": "2026-03-21T10:15:00"
  }
}
```

### 错误响应

| code | 场景 |
|------|------|
| 400 | `content` 为空或超过 2048 字符 |
| 400 | `rootCommentId` 指向的评论不存在或不属于当前笔记 |
| 400 | `replyCommentId` 指向的评论不存在 |
| 403 | 内容含高危敏感词（level=3）被拦截 |
| 404 | 笔记不存在或未发布 |
| 429 | 评论过于频繁（同一用户 60s 内超过 3 条） |

### 业务流程

```
客户端 POST /api/blog/10001/comments
          │
          ▼
┌──────────────────────┐
│ ① 笔记校验            │
│ · 笔记存在 且 status=1 │ ← 不存在/未发布 → 404
│ · content 非空 ≤2048  │ ← 校验失败 → 400
│ · rootCommentId 合法  │ ← 不存在/不属于此笔记 → 400
│ · replyCommentId 合法 │ ← 不存在 → 400
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│ ② 频率限制            │ ← 同一用户 60s 内 ≤ 3 条
│   Redis INCR + EXPIRE │    超限 → 429
│   key: comment:rate:  │
│   {userId}            │
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│ ③ 敏感词过滤          │ ← tb_sensitive_word
│ · level=3 → 拒绝发表  │    → 403
│ · level=2 → 替换为*** │
│ · level=1 → 记录日志   │
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│ ④ 插入评论记录         │ ← INSERT tb_blog_comments
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│ ⑤ 更新笔记评论数       │ ← UPDATE tb_blog
│    comment_count + 1  │
└──────────┬───────────┘
           │
```

```
客户端 PUT /api/blog/10001/favorite
          │
          ▼
┌─────────────────────────┐
│ ① 笔记校验              │ ← 不存在/deleted/status≠1 → 404
│ SELECT status, deleted   │

└────────────┬────────────┘
             │

┌─────────────────────────┐

│ WHERE user_id = 当前用户  │
│   AND biz_type = 2       │  ← 2=笔记（全局统一编码）

└────────────┬────────────┘
             │
        ┌────┴────┐
        │ 已存在?  │
        └────┬────┘
       是    │    否
        ▼    │    ▼
  ┌──────────┐  ┌───────────────┐
  │ deleted?  │  │ INSERT 收藏    │
  │ =0 取消   │  │ deleted=0     │
  │ =1 恢复   │  └──────┬────────┘
  └─────┬────┘         │
        │              │

        │       ┌──────┘
        │       │ count +1
        ▼       ▼
```

```sql
-- ① 校验笔记存在

-- ② 查询是否已收藏（含已取消的记录）
SELECT id, deleted FROM tb_favorite
WHERE user_id = #{userId} AND biz_type = 2 AND biz_id = #{blogId};

WHERE id = #{blogId} AND deleted = 0;

UPDATE tb_favorite SET deleted = 0, create_time = NOW()
WHERE user_id = #{userId} AND biz_type = 2 AND biz_id = #{blogId};

UPDATE tb_blog SET favorite_count = favorite_count + 1
WHERE id = #{blogId} AND deleted = 0;

-- ③-C 取消收藏（逻辑删除）
UPDATE tb_favorite SET deleted = 1
WHERE user_id = #{userId} AND biz_type = 2 AND biz_id = #{blogId};

UPDATE tb_blog SET favorite_count = GREATEST(favorite_count - 1, 0)
WHERE id = #{blogId} AND deleted = 0;
```

## 十三、关键设计决策

| 决策点 | 方案 | 理由 |
|--------|------|------|
| AI 摘要 | **独立接口 + 懒加载缓存 + 降级兜底** | 不阻塞主详情接口；LLM 故障时截取正文前100字 |
| 评论加载 | **顶级分页 + 子评论预加载前3条 + 展开接口** | 减少请求次数，展开更多时调用接口 8 |
| 评论排序 | **动态 SQL** 分别走 `hot`/`new` 两套索引 | 避免 CASE WHEN 导致全表 filesort |
| 子评论截取 | **窗口函数 ROW_NUMBER()** 每组取前3 | 避免拉取全量再截断，减少 IO |
| 收藏存储 | `tb_favorite` 逻辑删除 + `tb_blog.favorite_count` 冗余 | 与 like_count/comment_count 风格一致 |
| 关注计数 | **同步冗余**到 `tb_user_info`，同一事务 | 避免 COUNT(*)，写时维护，事务保证一致性 |
| 关注恢复 | 先查再决定 INSERT 或 UPDATE `deleted=0` | 避免唯一索引冲突 |
| 浏览记录 | **异步写入**，仅已登录用户 | 不影响详情接口响应速度 |
| 消息通知 | **异步 MQ** 发送（事务提交后） | 评论/关注后不阻塞主流程 |
| 评论频控 | Redis INCR + EXPIRE，60s 内 ≤ 3 条 | 防止刷评论 |
| 评论点赞 | 新增 `tb_comment_likes` 表 | 支持评论 isLiked 状态和去重 |
| biz_type | 全局统一编码（1商户 2笔记 3评论 4用户） | 避免跨表编码不一致 |
