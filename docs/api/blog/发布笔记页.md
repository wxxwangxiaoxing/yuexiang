## 二、业务类型编码说明

> 本文档涉及两种不同含义的 `type` 字段，需明确区分：

| 表.字段 | 含义 | 编码 | 说明 |
|---------|------|------|------|
| `tb_tag_relation.biz_type` | 关联目标类型 | **1=商户, 2=笔记** | 全局统一 biz_type 编码 |
| `tb_favorite.biz_type` | 收藏目标类型 | **1=商户, 2=笔记** | 全局统一 biz_type 编码 |
| `tb_tag.type` | 标签自身分类 | **0=商户标签, 1=笔记话题** | 标签池分类，与 biz_type 无关 |

> `tb_tag.type` 是标签自身的分类属性（"这个标签属于商户类还是笔记类"），与 `biz_type`（"这条关联记录指向商户还是笔记"）是不同概念。

## 四、接口 1 — 上传笔记图片

Page 5 已设计，此处补充**多图排序**场景。

### 基本信息

```
POST /api/upload/blog
Content-Type: multipart/form-data
Authorization: Bearer <token>
```

### 请求参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `file` | MultipartFile | **是** | 图片文件 |

### 校验规则

```
文件大小:   ≤ 5MB
文件类型:   image/jpeg, image/png, image/webp
单篇上限:   9张（前端控制上传入口，后端在发布时校验 images 数组长度）
```

### 成功响应

```json
{
  "code": 200,
  "msg": "ok",
  "data": {
    "filePath": "/imgs/blog/2025/06/uuid-abc123.jpg",
    "fullUrl": "https://cdn.yuexiang.com/imgs/blog/2025/06/uuid-abc123.jpg"
  }
}
```

### 错误响应

| code | 场景 |
|------|------|
| 400 | 文件为空 / 超过 5MB / 类型不合法 |
| 401 | 未登录 |

## 五、接口 2 — 发布笔记

### 基本信息

```
POST /api/blog
Content-Type: application/json
Authorization: Bearer <token>
```

### 请求体

```json
{
  "draftId": null,
  "shopId": 5001,
  "title": "望京最好吃的川菜，没有之一！",
  "content": "上周末和朋友去了望京新开的川味小馆...",
  "images": [
    "/imgs/blog/2025/06/a1.jpg",
    "/imgs/blog/2025/06/a2.jpg",
    "/imgs/blog/2025/06/a3.jpg"
  ],
  "tagIds": [1, 5, 12],
  "location": "望京SOHO"
}
```

### 参数校验

| 字段 | 类型 | 必填 | 校验规则 |
|------|------|------|----------|
| `draftId` | Long | 否 | 草稿ID（不为空时表示从草稿发布，校验草稿存在且属于当前用户） |
| `shopId` | Long | **是** | 商户存在且未删除 |
| `title` | String | **是** | 1~255 字符 |
| `content` | String | 否 | ≤ 10000 字符 |
| `images` | String[] | **是** | 1~9 张，路径格式合法 |
| `tagIds` | Long[] | 否 | 0~10 个，标签必须存在且 `type=1`（笔记话题） |
| `location` | String | 否 | ≤ 128 字符 |

### 成功响应

```json
{
  "code": 200,
  "msg": "发布成功，审核中",
  "data": {
    "blogId": 10088
  }
}
```

### 错误响应

| code | 场景 |
|------|------|
| 400 | title 为空或超长 / images 为空或超过 9 张 / shopId 不存在 / tagIds 中有不存在的标签 |
| 400 | draftId 指向的草稿不存在 / 不属于当前用户 / 状态不是草稿 |
| 403 | 标题或正文含高危敏感词（level=3）被拦截 |
| 401 | 未登录 |
| 429 | 发布频率限制（同一用户 5 分钟内最多发布 1 篇） |

### 业务流程

```
客户端 POST /api/blog
          │
          ▼
┌──────────────────────────┐
│ ① 参数校验                │
│ · title 非空 ≤255         │
│ · images 1~9张            │
│ · shopId 存在             │
│ · tagIds 合法 ≤10个       │
│ · draftId 合法（如有）     │
└────────────┬─────────────┘
             │
             ▼
┌──────────────────────────┐
│ ② 发布频率限制            │ ← Redis: blog:rate:{userId}
│   同一用户 5分钟内 ≤ 1篇  │    超限 → 429
└────────────┬─────────────┘
             │
             ▼
┌──────────────────────────┐
│ ③ 敏感词过滤              │ ← tb_sensitive_word
│ · title + content         │
│ · level=3 → 拦截返回403   │
│ · level=2 → 替换为 ***    │
│ · level=1 → 标记人工复审   │
└────────────┬─────────────┘
             │
             ▼
┌──────────────────────────────────────────┐
│ ④ 写入笔记（同一事务）                     │
│                                          │
│ 无 draftId：INSERT tb_blog (status=0)    │
│ 有 draftId：UPDATE 原草稿记录              │
│   · status: 3→0（草稿→待审核）            │
│   · 更新 title/content/images 等字段      │
│   · 校验 user_id 一致                     │
│                                          │
│ 处理标签关联：                              │
│ · 有 draftId → 先删除旧标签关联            │
│ · 批量 INSERT tb_tag_relation             │
│   biz_type = 2（笔记，全局统一编码）        │
│ · 更新 tb_tag.hot + 1                     │
└────────────┬─────────────────────────────┘
             │
             ▼
┌──────────────────────────┐
│ ⑤ 异步: 内容审核（MQ）    │
│ · 图片鉴黄 (AI Vision)    │
│ · 文本合规 (AI/规则)       │
│ · 通过 → status = 1       │
│ · 不通过 → status = 2     │
│ · 通知作者审核结果          │
│   tb_message type=1       │
└──────────────────────────┘
```

> ⚠️ 步骤 ④ 中的笔记写入 + 标签关联处理在同一个 `@Transactional` 中。步骤 ⑤ 为异步（事务提交后发 MQ 消息，由审核消费者处理）。

### 核心 SQL

```sql
-- 新发布（无 draftId）
INSERT INTO tb_blog
    (shop_id, user_id, title, content, images, location, status, created_by)
VALUES
    (#{shopId}, #{userId}, #{title}, #{content}, #{imagesJson},
     #{location}, 0, #{userId});

-- 从草稿发布（有 draftId）
UPDATE tb_blog
SET title = #{title}, content = #{content}, images = #{imagesJson},
    shop_id = #{shopId}, location = #{location},
    status = 0, update_time = NOW(), updated_by = #{userId}
WHERE id = #{draftId}
  AND user_id = #{userId}
  AND status = 3
  AND deleted = 0;

-- 从草稿发布时，先清除旧标签关联
DELETE FROM tb_tag_relation
WHERE biz_type = 2 AND biz_id = #{blogId};

-- 批量插入标签关联（biz_type=2 表示笔记）
INSERT INTO tb_tag_relation (tag_id, biz_type, biz_id)
VALUES
    (#{tagId1}, 2, #{blogId}),
    (#{tagId2}, 2, #{blogId}),
    (#{tagId3}, 2, #{blogId});

-- 更新标签热度
UPDATE tb_tag SET hot = hot + 1
WHERE id IN (#{tagIds}) AND deleted = 0;
```

## 六、接口 3 — 创建草稿

首次保存草稿时调用，返回新草稿 ID，后续更新使用接口 4。

### 基本信息

```
POST /api/blog/draft
Content-Type: application/json
Authorization: Bearer <token>
```

### 请求体

所有字段均为可选（允许部分填写的草稿）：

```json
{
  "shopId": 5001,
  "title": "望京最好吃的川菜",
  "content": "上周末和朋友去了...",
  "images": ["/imgs/blog/2025/06/a1.jpg"],
  "tagIds": [1, 5],
  "location": "望京SOHO"
}
```

### 参数校验（草稿宽松模式）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `title` | String | 否 | 草稿允许空标题，有值时 ≤ 255 字符 |
| `content` | String | 否 | 可部分填写，≤ 10000 字符 |
| `images` | String[] | 否 | 可不上传图片，有值时 ≤ 9 张 |
| `shopId` | Long | 否 | 可暂不关联商户，有值时校验存在 |
| `tagIds` | Long[] | 否 | 可无标签，有值时 ≤ 10 个 |
| `location` | String | 否 | ≤ 128 字符 |

### 成功响应

```json
{
  "code": 200,
  "msg": "草稿已保存",
  "data": {
    "blogId": 10089,
    "savedTime": "2026-03-21T10:30:00"
  }
}
```

### 错误响应

| code | 场景 |
|------|------|
| 400 | title 超长 / images 超过 9 张 / shopId 不存在 |
| 401 | 未登录 |
| 429 | 草稿保存过于频繁（同一用户 10 秒内最多 1 次） |

### 核心 SQL

```sql
-- 创建新草稿
INSERT INTO tb_blog
    (shop_id, user_id, title, content, images, location, status, created_by)
VALUES
    (#{shopId}, #{userId}, #{title}, #{content}, #{imagesJson},
     #{location}, 3, #{userId});
     -- status=3 表示草稿

-- 草稿也保存标签关联（如有 tagIds）
INSERT INTO tb_tag_relation (tag_id, biz_type, biz_id)
VALUES
    (#{tagId1}, 2, #{blogId}),
    (#{tagId2}, 2, #{blogId});
-- 注意：草稿阶段不更新 tb_tag.hot，发布时才更新
```

## 七、接口 4 — 更新草稿

更新已有草稿内容，前端自动保存或用户手动保存时调用。

### 基本信息

```
PUT /api/blog/draft/{id}
Content-Type: application/json
Authorization: Bearer <token>
```

### 路径参数

| 参数 | 类型 | 说明 |
|------|------|------|
| `id` | Long | 草稿ID |

### 请求体

与创建草稿相同结构，所有字段可选：

```json
{
  "shopId": 5001,
  "title": "望京最好吃的川菜",
  "content": "上周末和朋友去了...",
  "images": ["/imgs/blog/2025/06/a1.jpg"],
  "tagIds": [1, 5],
  "location": "望京SOHO"
}
```

### 成功响应

```json
{
  "code": 200,
  "msg": "草稿已保存",
  "data": {
    "blogId": 10089,
    "savedTime": "2026-03-21T10:35:00"
  }
}
```

### 错误响应

| code | 场景 |
|------|------|
| 400 | 参数校验不通过 |
| 403 | 草稿不属于当前用户 |
| 404 | 草稿不存在 / 已删除 / 状态不是草稿（status≠3） |
| 401 | 未登录 |
| 429 | 保存过于频繁（10 秒内最多 1 次） |

### 核心 SQL

```sql
-- 更新草稿内容
UPDATE tb_blog
SET title = #{title}, content = #{content}, images = #{imagesJson},
    shop_id = #{shopId}, location = #{location},
    updated_by = #{userId}, update_time = NOW()
WHERE id = #{id}
  AND user_id = #{userId}
  AND status = 3
  AND deleted = 0;

-- 更新草稿标签：先删后插（同一事务）
DELETE FROM tb_tag_relation
WHERE biz_type = 2 AND biz_id = #{blogId};

-- tagIds 非空时重新插入
INSERT INTO tb_tag_relation (tag_id, biz_type, biz_id)
VALUES
    (#{tagId1}, 2, #{blogId}),
    (#{tagId2}, 2, #{blogId});
```

## 八、接口 5 — AI 生成标题

根据正文内容，生成 3 个候选标题供用户选择。

### 基本信息

```
POST /api/ai/blog/title
Content-Type: application/json
Authorization: Bearer <token>
```

### 请求体

```json
{
  "content": "上周末和朋友去了望京新开的川味小馆，一进门就被浓浓的川味香气吸引...",
  "images": [
    "/imgs/blog/2025/06/a1.jpg",
    "/imgs/blog/2025/06/a2.jpg"
  ]
}
```

### 参数校验

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `content` | String | **是** | 正文内容，至少 10 字 |
| `images` | String[] | 否 | 图片路径（可用于多模态分析，当前版本未启用） |

### 成功响应

```json
{
  "code": 200,
  "msg": "ok",
  "data": {
    "titles": [
      "望京最好吃的川菜，没有之一！",
      "川味小馆探店｜水煮鱼和毛血旺绝了",
      "人均85的宝藏川菜馆，闺蜜聚餐首选"
    ],
    "fallback": false
  }
}
```

| 字段 | 说明 |
|------|------|
| `titles` | 候选标题列表（正常 3 个，降级时 1 个） |
| `fallback` | 是否为降级内容 |

### 错误响应

| code | 场景 |
|------|------|
| 400 | content 少于 10 字 |
| 401 | 未登录 |
| 429 | AI 使用次数达到每日上限 |

### 业务流程

```
客户端 POST /api/ai/blog/title
          │
          ▼
┌───────────────────────────┐
│ ① 参数校验                 │
│ · content ≥ 10字           │
└─────────────┬─────────────┘
              │
              ▼
┌───────────────────────────┐
│ ② 频率限制                 │ ← Redis: ai:limit:title:{userId}
│ · 本接口 ≤ 20次/日         │     + ai:limit:total:{userId}
│ · 全部AI ≤ 50次/日         │    超限 → 429
└─────────────┬─────────────┘
              │
              ▼
┌───────────────────────────────────────────────┐
│ ③ 构建 Prompt                                  │
│                                               │
│ System: 你是探店笔记标题生成助手，根据内容       │
│ 生成3个吸引人的标题，每个≤30字，风格活泼，       │
│ 适合年轻用户。仅返回标题，每行一个。             │
│                                               │
│ User: {content}                               │
└─────────────┬─────────────────────────────────┘
              │
              ▼
┌───────────────────────────┐
│ ④ 调用 LLM API (超时10s)   │
│ (DeepSeek / GPT / 通义)    │
│                           │
│ · model: deepseek-chat    │
│ · temperature: 0.8        │
│ · max_tokens: 200         │
└─────────────┬─────────────┘
         ┌────┴────┐
         │ 成功?    │
         └────┬────┘
        是    │    否（超时/异常/熔断）
         ▼    │    ▼
   ┌──────────┐  ┌──────────────────────┐
   │ 解析响应  │  │ 降级: 截取正文前20字   │
   │ 分割标题  │  │ 加"探店分享"作为标题  │
   │ 敏感词过滤│  │ fallback=true         │
   │ 长度截断  │  │ titles 返回 1 个       │
   └──────────┘  └──────────────────────┘
         │              │
         └──────┬───────┘
                ▼
   ┌───────────────────────────┐
   │ ⑤ INCR Redis 计数器       │
   │ ⑥ 记录AI调用日志          │ ← tb_operation_log
   │    module=AI               │
   │    action=GENERATE_TITLE   │
   └───────────────────────────┘
```

**AI 降级策略**：
- LLM 调用超时设为 **10 秒**
- 连续失败 **3 次**后触发熔断，30 秒内直接走降级
- 降级时返回正文前 20 字 + "探店分享"作为单个候选标题
- `fallback=true` 标记降级内容

**LLM 响应解析容错**：
- LLM 返回非预期格式时（如 JSON 而非纯文本），尝试多种分割方式（换行符、逗号、数字编号）
- 解析失败时走降级逻辑，不抛异常

## 九、接口 6 — AI 润色内容

优化文笔、修正语病、增强表达力，保持原意。

### 基本信息

```
POST /api/ai/blog/polish
Content-Type: application/json
Authorization: Bearer <token>
```

### 请求体

```json
{
  "content": "上周末去了望京一个川菜馆，味道还不错，环境也行，推荐大家去",
  "style": "lively"
}
```

### 参数校验

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `content` | String | **是** | 原始正文，10~10000 字 |
| `style` | String | 否 | 风格偏好（默认 `lively`） |

### style 可选值

| 值 | 说明 |
|---|------|
| `lively` | 活泼生动（默认） |
| `literary` | 文艺清新 |
| `professional` | 专业客观 |
| `humorous` | 幽默风趣 |

### 成功响应

```json
{
  "code": 200,
  "msg": "ok",
  "data": {
    "original": "上周末去了望京一个川菜馆，味道还不错，环境也行，推荐大家去",
    "polished": "上周末和闺蜜打卡了望京新开的川味小馆🌶️ 一进门就被麻辣鲜香勾住了魂！环境温馨有格调，特别适合朋友小聚～ 强烈安利给你们！",
    "changes": [
      {
        "type": "enhance",
        "description": "增加感官描写和emoji，增强画面感"
      },
      {
        "type": "detail",
        "description": "补充场景细节，让读者身临其境"
      }
    ],
    "fallback": false
  }
}
```

### 错误响应

| code | 场景 |
|------|------|
| 400 | content 少于 10 字或超过 10000 字 / style 值不合法 |
| 401 | 未登录 |
| 429 | AI 使用次数达到每日上限 |

### Prompt 构建


### 请求参数（Query）

| 参数 | 类型 | 必填 | 默认 | 说明 |

```json
        "address": "望京SOHO B座1层",
        "avgPrice": 8500,
        "score": 4.5
      },
      {
        "id": 5023,

        "typeName": "川菜",
        "address": "国贸商城B1层",
        "avgPrice": 9500,

      }
```

### 错误响应

| code | 场景 |
|------|------|
| 400 | keyword 为空或超过 64 字 |
| 401 | 未登录 |

### 核心 SQL

```sql
-- 基础方案：LIKE 模糊搜索
-- 注意：前缀 LIKE 'keyword%' 可走索引，'%keyword%' 不走索引
-- 数据量大时（>10万商户）建议改用 Elasticsearch 全文检索
SELECT s.id, s.name, s.address, s.avg_price, s.score,
       st.name AS type_name
FROM tb_shop s
JOIN tb_shop_type st ON st.id = s.type_id AND st.deleted = 0
WHERE s.deleted = 0
  AND s.name LIKE CONCAT('%', #{keyword}, '%')
ORDER BY s.score DESC
LIMIT #{offset}, #{size};
```

> **性能说明**：当前使用 `LIKE '%keyword%'` 无法走索引，适用于商户数量 < 10 万的场景。若商户数据量增长，应接入 Elasticsearch 做全文检索，搜索接口路由到 ES 查询。

## 十五、AI 通用降级策略

> 所有 AI 接口共享以下降级机制：

| 降级条件 | 行为 |
|----------|------|
| 单次 LLM 调用超时（>10s） | 走降级逻辑，返回 `fallback=true` |
| 连续失败 3 次 | 触发熔断，30 秒内直接走降级 |
| LLM 返回格式异常 | 尝试多种解析方式，全部失败走降级 |
| 降级内容 | **不计入** AI 调用次数 |

| 接口 | 降级内容 |
|------|----------|
| AI 生成标题 | 截取正文前 20 字 + "探店分享"作为单个候选标题 |
| AI 润色 | 返回原文不修改，`polished` = 原文 |
| AI 扩写 | 返回原文不修改，`expanded` = 原文 |
| AI 打标签 | jieba 分词提取 TOP5 名词，再匹配标签库 |

## 十七、关键设计决策

| 决策点 | 方案 | 理由 |
|--------|------|------|
| AI 接口独立拆分 | 标题/润色/扩写/标签 **4个独立接口** | 按需调用，不浪费 token；前端灵活组合 |
| AI 频率限制 | **Redis 计数器** + 单接口限制 + 全局限制 | 控制 LLM 成本，防滥用 |
| AI 降级 | **超时+熔断+降级兜底** + `fallback` 标记 | LLM 不可用时用户仍可使用基础功能 |
| AI 响应解析 | **多种格式尝试** + 解析失败走降级 | LLM 输出格式不可控，需容错 |
| 草稿机制 | `status=3` 复用 `tb_blog` 表，**POST 创建 + PUT 更新** | RESTful 规范；无需额外建表 |
| 草稿转发布 | 发布接口含 `draftId` 参数，复用原记录 | 避免重复数据 |
| 草稿标签 | 草稿阶段保存标签关联但**不更新热度** | 发布时才计入热度，避免草稿污染 |
| 图片先传后发 | **两步操作**：上传返回路径 → 发布携带路径 | 前端预览体验好；避免大表单超时 |
| 敏感词检查 | **发布时同步检查**（高危词） + **异步 AI 复审**（低危词） | 高危词即时拦截；低危词不阻塞发布 |
| 内容审核 | **事务提交后发 MQ**，审核消费者异步处理 | 不阻塞发布流程；审核结果通过消息通知 |
| 商户搜索 | LIKE 模糊 + 评分排序，预留 ES 升级路径 | 初期简单实现，数据量大时切 ES |
| biz_type 统一 | `tb_tag_relation.biz_type = 2` 表示笔记 | 与 `tb_favorite`、`tb_browse_history` 等表全局一致 |
| 发布频率 | 同一用户 5 分钟内最多 1 篇 | 防止刷笔记 |
| 草稿保存频率 | 同一用户 10 秒内最多 1 次 | 前端自动保存场景，防止高频写入 |
