## 错误码体系

> 所有秒杀接口统一使用以下错误码。

| 错误码  | HTTP 状态码 | 说明             | 前端处理                |
|---------|------------|------------------|------------------------|
| `200`   | 200        | 成功             | 正常处理                |
| `40010` | 400        | 秒杀尚未开始      | 显示倒计时              |
| `40011` | 400        | 秒杀已结束        | 按钮置灰 + 提示         |
| `40012` | 400        | 库存不足/已抢光    | 按钮置灰 + 提示"已抢光" |
| `40013` | 400        | 重复购买          | 提示"已抢过"+ 跳转订单  |
| `40014` | 400        | 券不存在/已下架    | 提示 + 刷新列表         |
| `40015` | 429        | 请求过于频繁       | 提示稍后重试            |
| `40016` | 400        | 下单处理中请勿重复提交 | 等待结果            |
| `40100` | 401        | 未登录            | 跳转登录页              |
| `50000` | 500        | 服务器内部错误     | 通用错误提示            |
| `50010` | 503        | 系统繁忙(降级)     | 提示稍后重试            |

标准响应格式：

```json
{
  "code": 40012,
  "msg": "很抱歉，已被抢光",
  "data": null
}
```

## 接口 1：获取服务器时间

**用途：** 前端与服务端时间同步，保证倒计时精准。

```plain
GET /api/seckill/server-time
```

### 请求参数

无

### 响应

```json
{
  "code": 200,
  "msg": "ok",
  "data": {
    "serverTime": 1750838400000
  }
}
```

| 字段         | 类型 | 说明                     |
|-------------|------|-------------------------|
| `serverTime`| Long | 服务器当前时间戳（毫秒）   |

### 前端时间同步逻辑

```plain
// 计算客户端与服务端的时间偏移
offset = serverTime - clientTime

// 所有倒计时使用修正后的时间
correctedNow = Date.now() + offset
remaining = session.endTime - correctedNow
```

### 字段说明

| 字段           | 类型    | 说明                                             |
|---------------|---------|--------------------------------------------------|
| `serverTime`  | Long    | 服务器当前时间戳（毫秒），前端据此计算偏移           |
| `sessionId`   | Long    | 场次ID                                            |
| `title`       | String  | 场次名称                                          |
| `beginTime`   | Long    | 场次开始时间（毫秒时间戳）                          |
| `endTime`     | Long    | 场次结束时间（毫秒时间戳）                          |
| `status`      | Integer | 状态：**0**=即将开始 **1**=抢购中 **2**=已结束      |
| `statusDesc`  | String  | 状态文案                                          |
| `voucherCount`| Integer | 该场次券数量                                       |

### 场次状态判定逻辑（后端计算）

```plain
serverTime < beginTime                  →  0 即将开始
beginTime ≤ serverTime < endTime        →  1 抢购中
serverTime ≥ endTime                    →  2 已结束
```

### 数据来源 SQL

```sql
SELECT
    ss.id            AS session_id,
    ss.title,
    ss.begin_time,
    ss.end_time,
    COUNT(sv.voucher_id) AS voucher_count
FROM tb_seckill_session ss
LEFT JOIN tb_seckill_voucher sv
    ON sv.session_id = ss.id AND sv.deleted = 0
WHERE ss.date = #{date}
  AND ss.status = 1
  AND ss.deleted = 0
GROUP BY ss.id
ORDER BY ss.begin_time ASC;
```

### 库存百分比与秒杀状态判定（后端计算）

```java
// 库存百分比（防除零）
stockPercent = totalStock > 0
    ? (int)(remainStock * 100.0 / totalStock)
    : 0;

// 秒杀状态
if (now < beginTime)          → 0 未开始
else if (now >= endTime)      → 2 已结束
else if (remainStock <= 0)    → 3 已抢光
else                          → 1 进行中
```

### 库存数据来源（Redis 优先）

```plain
获取 remainStock 的优先级:
┌───────────────────────────────────────────────┐
│ P1: Redis GET seckill:stock:{voucherId}       │
│     → 活动期间 Redis 有预热数据，直接返回       │
│                                               │
│ P2: 降级读 DB                                  │
│     SELECT stock FROM tb_seckill_voucher      │
│     WHERE voucher_id = #{voucherId}           │
│     → Redis 不可用或未预热时降级               │
└───────────────────────────────────────────────┘
```

### 数据来源 SQL

```sql
SELECT
    v.id           AS voucher_id,
    v.shop_id,
    s.name         AS shop_name,
    s.images       AS shop_images,      -- JSON 数组，Java 侧取第一张
    v.title,
    v.sub_title,
    v.pay_value,
    v.actual_value,
    sv.total_stock,
    sv.stock       AS remain_stock,     -- DB 库存（Redis 不可用时降级使用）
    sv.begin_time,
    sv.end_time
FROM tb_voucher v
INNER JOIN tb_seckill_voucher sv ON sv.voucher_id = v.id
INNER JOIN tb_shop s             ON s.id = v.shop_id
WHERE sv.session_id = #{sessionId}
  AND v.type = 1                        -- 秒杀券
  AND v.status = 0                      -- 上架中
  AND v.deleted = 0
  AND sv.deleted = 0
  AND s.deleted = 0
ORDER BY sv.stock DESC, v.id ASC        -- 有库存的排前面
LIMIT #{offset}, #{pageSize};

-- 总数查询
SELECT COUNT(1)
FROM tb_voucher v
INNER JOIN tb_seckill_voucher sv ON sv.voucher_id = v.id
WHERE sv.session_id = #{sessionId}
  AND v.type = 1 AND v.status = 0
  AND v.deleted = 0 AND sv.deleted = 0;
```

SELECT COUNT(1)
FROM tb_voucher_order
WHERE voucher_id = #{voucherId}
  AND user_id = #{userId}
  AND status IN (0, 1, 2)    -- 未支付/已支付/已使用 均视为已占
  AND deleted = 0;

-- 未登录   → hasBought = false（不查询）

### ⚡ 后端核心流程（高并发安全方案）

```plain
┌──────────────────────────────────────────────────────────────┐
│                      秒杀下单完整流程                          │
│                                                              │
│  ┌────────────────────────────────────────────────────────┐  │
│  │ ① 基础校验                                              │  │
│  │                                                        │  │
│  │   · 参数校验: voucherId > 0                             │  │
│  │   · 登录校验: Token 有效，提取 userId                    │  │
│  │   · 用户限流: Redis 滑动窗口                             │  │
│  │     Key: rate:seckill:user:{userId}                    │  │
│  │     每秒 ≤ 5 次（读 tb_sys_config 配置）                 │  │
│  │     超限 → 返回 40015                                   │  │
│  └────────────────────────┬───────────────────────────────┘  │
│                           │                                  │
│                           ▼                                  │
│  ┌────────────────────────────────────────────────────────┐  │
│  │ ② 时间校验（Java 侧快速失败）                            │  │
│  │                                                        │  │
│  │   从 Redis Hash 读取券的时间信息:                         │  │
│  │   HGETALL seckill:voucher:{voucherId}                  │  │
│  │                                                        │  │
│  │   if 不存在 → 降级查 DB，仍不存在 → 返回 40014           │  │
│  │   if now < beginTime → 返回 40010（未开始）              │  │
│  │   if now ≥ endTime   → 返回 40011（已结束）              │  │
│  └────────────────────────┬───────────────────────────────┘  │
│                           │                                  │
│                           ▼                                  │
│  ┌────────────────────────────────────────────────────────┐  │
│  │ ③ Redis Lua 原子操作（库存扣减 + 一人一单）               │  │
│  │                                                        │  │
│  │   KEYS[1] = seckill:stock:{voucherId}   -- 库存        │  │
│  │   KEYS[2] = seckill:order:{voucherId}   -- 已购用户集合 │  │
│  │   ARGV[1] = userId                                     │  │
│  │                                                        │  │
│  │   ┌──────────────────────────────────────────────────┐ │  │
│  │   │ -- 1. 判断库存                                    │ │  │
│  │   │ local stock = tonumber(                           │ │  │
│  │   │     redis.call('get', KEYS[1]))                   │ │  │
│  │   │ if stock == nil or stock <= 0 then                │ │  │
│  │   │     return 1  -- 库存不足                          │ │  │
│  │   │ end                                               │ │  │
│  │   │                                                   │ │  │
│  │   │ -- 2. 判断是否重复下单                              │ │  │
│  │   │ if redis.call('sismember',                        │ │  │
│  │   │        KEYS[2], ARGV[1]) == 1 then                │ │  │
│  │   │     return 2  -- 重复下单                          │ │  │
│  │   │ end                                               │ │  │
│  │   │                                                   │ │  │
│  │   │ -- 3. 扣库存 + 记录用户                            │ │  │
│  │   │ redis.call('incrby', KEYS[1], -1)                 │ │  │
│  │   │ redis.call('sadd', KEYS[2], ARGV[1])              │ │  │
│  │   │ return 0  -- 成功                                 │ │  │
│  │   └──────────────────────────────────────────────────┘ │  │
│  │                                                        │  │
│  │   Lua 返回值:                                           │  │
│  │     0 → 成功，继续                                      │  │
│  │     1 → 返回 40012（库存不足）                           │  │
│  │     2 → 返回 40013（重复购买）                           │  │
│  └────────────────────────┬───────────────────────────────┘  │
│                           │ Lua 返回 0                       │
│                           ▼                                  │
│  ┌────────────────────────────────────────────────────────┐  │
│  │ ④ 生成订单ID + 发送消息到 MQ                             │  │
│  │                                                        │  │
│  │   orderId  = 雪花算法生成                                │  │
│  │   orderNo  = "YX" + yyyyMMdd + 序列号                   │  │
│  │                                                        │  │
│  │   MQ Message:                                          │  │
│  │   Topic: seckill.order.create                          │  │
│  │   Body: {                                              │  │
│  │     orderId,                                           │  │
│  │     orderNo,                                           │  │
│  │     voucherId,                                         │  │
│  │     userId,                                            │  │
│  │     timestamp                                          │  │
│  │   }                                                    │  │
│  │                                                        │  │
│  │   ⚠ MQ 发送失败补偿:                                    │  │
│  │   ┌──────────────────────────────────────────────────┐ │  │
│  │   │ try {                                            │ │  │
│  │   │     mqProducer.send(message);                    │ │  │
│  │   │ } catch (Exception e) {                          │ │  │
│  │   │     // 回滚 Redis: 恢复库存 + 移除用户记录         │ │  │
│  │   │     redis.eval(rollbackLua,                      │ │  │
│  │   │         KEYS=[stock_key, order_key],             │ │  │
│  │   │         ARGV=[userId]);                          │ │  │
│  │   │     // rollbackLua:                              │ │  │
│  │   │     //   redis.call('incrby', KEYS[1], 1)        │ │  │
│  │   │     //   redis.call('srem', KEYS[2], ARGV[1])    │ │  │
│  │   │     log.error("MQ发送失败，已回滚Redis", e);       │ │  │
│  │   │     throw new BizException(50010, "系统繁忙");    │ │  │
│  │   │ }                                                │ │  │
│  │   └──────────────────────────────────────────────────┘ │  │
│  └────────────────────────┬───────────────────────────────┘  │
│                           │                                  │
│                           ▼                                  │
│  ┌────────────────────────────────────────────────────────┐  │
│  │ ⑤ 立即返回订单ID给前端                                   │  │
│  │                                                        │  │
│  │   { orderId, orderNo }                                 │  │
│  │   此时 DB 尚未写入，前端进入轮询（接口6）                  │  │
│  └────────────────────────────────────────────────────────┘  │
│                                                              │
│ ═══════════════ 以下为异步处理（MQ 消费者） ══════════════════ │
│                                                              │
│  ┌────────────────────────────────────────────────────────┐  │
│  │ ⑥ MQ 消费者：异步持久化                                  │  │
│  │                                                        │  │
│  │  a. 再次校验（防御性编程）                                │  │
│  │     · 查 DB 确认券存在 + 状态正常                        │  │
│  │     · 查 DB 确认一人一单                                 │  │
│  │                                                        │  │
│  │  b. DB 扣减库存（CAS + UNSIGNED 双保险）                 │  │
│  │     UPDATE tb_seckill_voucher                          │  │
│  │     SET stock = stock - 1,                             │  │
│  │         update_time = NOW()                            │  │
│  │     WHERE voucher_id = #{voucherId}                    │  │
│  │       AND stock > 0;                                   │  │
│  │                                                        │  │
│  │     影响行数 = 0 → 库存不足                              │  │
│  │     → 回滚 Redis + 记录异常日志                          │  │
│  │                                                        │  │
│  │  c. 创建订单                                            │  │
│  │     INSERT INTO tb_voucher_order                       │  │
│  │       (id, order_no, user_id, voucher_id,              │  │
│  │        pay_type, status, create_time)                   │  │
│  │     VALUES                                             │  │
│  │       (#{orderId}, #{orderNo}, #{userId},              │  │
│  │        #{voucherId}, 0, 0, NOW());                     │  │
│  │                                                        │  │
│  │  d. 发送延迟消息（超时取消用）                             │  │
│  │     Topic: seckill.order.timeout                       │  │
│  │     Delay: 30分钟（读 tb_sys_config 配置）               │  │
│  │     Body: { orderId, voucherId, userId }               │  │
│  │                                                        │  │
│  │  e. 库存归零检测                                        │  │
│  │     if (剩余库存 == 0):                                 │  │
│  │       → 广播售罄事件（可选 WebSocket 通知前端）            │  │
│  │       → 更新 tb_voucher.status = 1（下架）可选            │  │
│  │                                                        │  │
│  │  f. 记录操作日志                                        │  │
│  │     INSERT tb_operation_log                            │  │
│  │     (module='SECKILL', action='CREATE_ORDER', ...)     │  │
│  └────────────────────────────────────────────────────────┘  │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### Redis 数据结构（预热阶段写入）

```plain
═══════════════════════════════════════════════════════════
  每个秒杀券预热写入以下 Key
═══════════════════════════════════════════════════════════

# 1. 库存 (String)
SET seckill:stock:{voucherId} {totalStock}

# 2. 已购用户集合 (Set) — 初始为空
DEL seckill:order:{voucherId}

# 3. 券信息缓存 (Hash) — 含时间信息供校验
HMSET seckill:voucher:{voucherId}
  voucherId   {voucherId}
  sessionId   {sessionId}
  shopName    {shopName}
  title       {title}
  payValue    {payValue}
  actualValue {actualValue}
  totalStock  {totalStock}
  beginTime   {beginTime毫秒}
  endTime     {endTime毫秒}

# 4. 统一设置过期时间（活动结束后2小时自动清理）
EXPIREAT seckill:stock:{voucherId}   {endTimestamp + 7200}
EXPIREAT seckill:order:{voucherId}   {endTimestamp + 7200}
EXPIREAT seckill:voucher:{voucherId} {endTimestamp + 7200}
```

## 接口 6：查询秒杀订单结果

**用途：** 前端下单后轮询，确认异步订单是否创建成功。

```plain
GET /api/seckill/orders/{orderId}
Authorization: Bearer <token>
```

### 路径参数

| 参数      | 类型 | 必填 | 说明                  |
|----------|------|------|-----------------------|
| `orderId`| Long | 是   | 接口 5 返回的订单 ID   |

### 响应 — 订单已创建（待支付）

```json
{
  "code": 200,
  "msg": "ok",
  "data": {
    "orderId": 1925307648123904,
    "orderNo": "YX20260325000042",
    "voucherId": 1001,
    "shopName": "川味小馆",
    "title": "满100减20代金券",
    "payValue": 990,
    "status": 0,
    "statusDesc": "待支付",
    "createTime": 1750835000000,
    "payDeadline": 1750836800000
  }
}
```

### 响应 — 订单处理中（MQ 消费者尚未完成）

```json
{
  "code": 200,
  "msg": "ok",
  "data": {
    "orderId": 1925307648123904,
    "status": -1,
    "statusDesc": "处理中，请稍候"
  }
}
```

### 字段说明

| 字段          | 类型    | 说明                                                                                 |
|--------------|---------|--------------------------------------------------------------------------------------|
| `orderId`    | Long    | 订单ID                                                                                |
| `orderNo`    | String  | 订单编号（DB 写入后才有）                                                                |
| `status`     | Integer | **-1**=处理中(DB未写入) **0**=待支付 **1**=已支付 **2**=已使用 **3**=已退款 **4**=已取消 |
| `statusDesc` | String  | 状态文案                                                                               |
| `payDeadline`| Long    | 支付截止时间（毫秒），超时自动取消。读 `tb_sys_config` 配置，默认创建后30分钟              |

> **`status = -1` 说明：** 这是一个虚拟状态，表示 Redis 预扣成功、MQ 消息已发送，但 MQ 消费者尚未将订单写入 `tb_voucher_order` 表。此时查询 DB 找不到该订单，返回 -1 告知前端继续轮询。

### 查询逻辑

```plain
┌─────────────────────────────────────────────────┐
│ 查询 tb_voucher_order WHERE id = #{orderId}     │
│   AND user_id = #{userId}                       │
│                                                 │
│ ├─ 查到记录 → 返回完整订单信息                    │
│ │   status 直接取 DB 字段值                      │
│ │   payDeadline = createTime + 配置超时分钟数     │
│ │                                               │
│ └─ 未查到 → 进一步判断                           │
│     │                                           │
│     ├─ Redis SISMEMBER                          │
│     │  seckill:order:{voucherId} userId          │
│     │                                           │
│     ├─ 存在(=Redis预扣成功) → status = -1        │
│     │  "处理中，请稍候"                           │
│     │                                           │
│     └─ 不存在 → 订单不存在(404)                  │
│        "订单不存在或已失效"                       │
└─────────────────────────────────────────────────┘
```

### 前端轮询策略

```javascript
// 指数退避轮询
const INITIAL_INTERVAL = 500;    // 首次 500ms
const MAX_INTERVAL = 3000;       // 最大 3s
const MAX_RETRY = 20;            // 最多轮询 20 次

function pollOrder(orderId, retry = 0, interval = INITIAL_INTERVAL) {
  if (retry >= MAX_RETRY) {
    showToast('网络繁忙，请到「我的订单」中查看');
    return;
  }

  GET(`/api/seckill/orders/${orderId}`).then(res => {
    const { status } = res.data;

    switch (status) {
      case -1:
        // 仍在处理中，继续轮询
        const nextInterval = Math.min(interval * 1.5, MAX_INTERVAL);
        setTimeout(() => pollOrder(orderId, retry + 1, nextInterval), interval);
        break;

      case 0:
        // 待支付，跳转支付页
        navigateTo(`/pages/pay/index?orderId=${orderId}`);
        break;

      case 4:
        // 已取消（超时等）
        showToast('订单已取消');
        break;

      default:
        // 其他状态
        showResult(res.data);
    }
  }).catch(() => {
    // 网络错误，延迟重试
    setTimeout(() => pollOrder(orderId, retry + 1, MAX_INTERVAL), MAX_INTERVAL);
  });
}
```

## 超时未支付自动取消

```plain
┌──────────────────────────────────────────────────────────┐
│              超时自动取消 + 库存回补                        │
│                                                          │
│ ┌─ 方案 A: 延迟队列（推荐）──────────────────────────┐    │
│ │                                                    │    │
│ │ MQ 消费者创建订单时（步骤 ⑥-d），发送延迟消息:       │    │
│ │   Topic: seckill.order.timeout                     │    │
│ │   Delay: 30 分钟（读 tb_sys_config）                │    │
│ │   Body: { orderId, voucherId, userId }             │    │
│ │                                                    │    │
│ │ 延迟消息到达后，超时消费者处理:                       │    │
│ │                                                    │    │
│ │   ① 查询订单状态                                    │    │
│ │      SELECT status FROM tb_voucher_order           │    │
│ │      WHERE id = #{orderId} AND deleted = 0         │    │
│ │                                                    │    │
│ │   ② 判断是否需要取消                                │    │
│ │      status = 0（待支付） → 执行取消                 │    │
│ │      status ≠ 0 → 忽略（已支付/已取消等）             │    │
│ │                                                    │    │
│ │   ③ 取消订单（DB）                                  │    │
│ │      UPDATE tb_voucher_order                       │    │
│ │      SET status = 4, update_time = NOW()           │    │
│ │      WHERE id = #{orderId} AND status = 0;         │    │
│ │                                                    │    │
│ │   ④ 回补 DB 库存                                   │    │
│ │      UPDATE tb_seckill_voucher                     │    │
│ │      SET stock = stock + 1, update_time = NOW()    │    │
│ │      WHERE voucher_id = #{voucherId};              │    │
│ │                                                    │    │
│ │   ⑤ 回补 Redis 库存 + 移除用户占位                  │    │
│ │      INCRBY seckill:stock:{voucherId} 1            │    │
│ │      SREM   seckill:order:{voucherId} {userId}     │    │
│ │      (仅在 Redis Key 未过期时执行)                   │    │
│ │                                                    │    │
│ │   ⑥ 记录操作日志                                    │    │
│ │      INSERT tb_operation_log                       │    │
│ │      (module='SECKILL', action='TIMEOUT_CANCEL')   │    │
│ └────────────────────────────────────────────────────┘    │
│                                                          │
│ ┌─ 方案 B: 定时扫描（备选）─────────────────────────────┐  │
│ │                                                      │  │
│ │ Cron: 每分钟执行                                      │  │
│ │                                                      │  │
│ │ SELECT id, voucher_id, user_id                       │  │
│ │ FROM tb_voucher_order                                │  │
│ │ WHERE status = 0                                     │  │
│ │   AND create_time < DATE_SUB(NOW(),                  │  │
│ │       INTERVAL #{payTimeoutMin} MINUTE)              │  │
│ │   AND deleted = 0                                    │  │
│ │ LIMIT 100;                                           │  │
│ │                                                      │  │
│ │ → 批量执行上述 ③④⑤⑥ 步骤                            │  │
│ └──────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

## 接口限流与安全策略

| 策略           | 层级     | 实现方式                             | 参数                              |
|---------------|---------|-------------------------------------|----------------------------------|
| **全局限流**   | 网关     | Sentinel / Nginx `limit_req`        | 秒杀下单 QPS ≤ 1000（配置化）      |
| **IP 限流**    | Nginx   | `limit_req zone=seckill burst=50`   | 同一 IP 每秒 ≤ 50 次              |
| **用户限流**   | 应用层   | Redis 滑动窗口                       | 同一用户每秒 ≤ 5 次（配置化）       |
| **按钮防重**   | 前端+后端 | 前端节流 2s + 后端 Redis SETNX       | `seckill:btn:{userId}:{voucherId}` TTL=3s |
| **一人一单**   | Redis+DB | Redis Set 原子判断 + DB 查询双校验    | 贯穿 Lua 脚本 + MQ 消费者          |
| **库存防超卖** | Redis+DB | Redis Lua 原子扣减 + DB `stock > 0` CAS + `UNSIGNED` 兜底 | 三层防线 |
| **黑名单**     | 网关     | 异常行为检测 → 自动拉黑               | 1分钟内 > 100次请求 → 拉黑10分钟    |
| **请求签名**   | 前端+后端 | 时间戳 + 参数 + Secret 签名          | 防接口重放攻击                      |

### 按钮防重详细实现

```plain
前端:
┌───────────────────────────────────────────┐
│ 点击"立即抢购" → 立即禁用按钮 + loading     │
│ 无论成功失败 → 2秒后恢复按钮                 │
│ (防止网络延迟导致重复点击)                    │
└───────────────────────────────────────────┘

后端:
┌───────────────────────────────────────────┐
│ 步骤①之后，Lua脚本之前:                     │
│                                           │
│ String lockKey = "seckill:btn:"           │
│   + userId + ":" + voucherId;             │
│ Boolean acquired = redis.opsForValue()    │
│   .setIfAbsent(lockKey, "1",              │
│     Duration.ofSeconds(3));               │
│                                           │
│ if (!acquired)                            │
│   → 返回 40016 "处理中请勿重复提交"         │
│                                           │
│ // 注意：此锁在 Lua 执行完后自动过期(3s)     │
│ // 不需要手动释放                           │
└───────────────────────────────────────────┘
```

## 关键设计决策

| 决策点          | 方案                                | 理由                                                       |
|----------------|------------------------------------|------------------------------------------------------------|
| 库存存储        | **Redis String + DB UNSIGNED**      | Redis 承接高并发读写；DB UNSIGNED 字段级兜底防超卖             |
| 一人一单        | **Redis Set + DB 查询 双校验**       | Redis 原子操作拦截大部分并发；DB 最终一致性保障                 |
| 原子操作        | **Redis Lua 脚本**                  | 库存扣减 + 重复判断必须原子执行，避免竞态条件                    |
| 异步下单        | **MQ 异步持久化**                    | 秒杀瞬间高并发，DB 是瓶颈；Redis 预扣 + MQ 削峰填谷            |
| MQ 失败补偿     | **Lua 回滚脚本**                    | MQ 发送失败立即回滚 Redis，保证库存不丢失                       |
| 订单结果查询    | **前端指数退避轮询**                  | 比 WebSocket 实现简单；指数退避减少服务端压力                    |
| 超时取消        | **MQ 延迟消息 + 定时扫描兜底**        | 延迟消息精准；定时扫描兜底处理漏网之鱼                          |
| 库存回补        | **DB + Redis 双回补**                | 超时取消后 DB 和 Redis 都要恢复，保持数据一致                    |
| 时间校验位置    | **Java 侧快速失败**                  | 时间校验不需要原子性，Java 侧校验可快速返回，减少 Redis 压力      |
| 场次管理        | **独立 `tb_seckill_session` 表**     | 场次与券分离，支持灵活配置；避免按时间范围模糊匹配的不确定性       |
| 库存实时性      | **列表页读 Redis，降级读 DB**         | Redis 库存实时性高；DB 降级保证可用性                           |
| 按钮防重        | **前端节流 + 后端 SETNX**             | 双层防护；SETNX 3秒自动过期，不阻塞正常重试                     |
| 配置化          | **频率限制、超时时间等存 `tb_sys_config`** | 运营可在后台调整，无需重启服务                              |
| 预热机制        | **定时任务自动 + 手动触发**            | 自动预热减少人工操作；手动触发用于紧急场景                       |
| 虚拟状态 -1     | **前端感知 DB 未写入状态**             | 异步下单的必然产物；明确告知前端继续轮询而非报错                  |
