# 商户详情页 — 后端接口设计

***

## 〇、全局约定

```plain
金额约定：
  - 数据库：统一以"分"存储（INT / BIGINT）
  - 接口响应：统一以"元"返回（Double，保留两位小数）
  - 转换位置：Service 层 VO 组装时统一 ÷ 100.0
  - 示例：DB 存 990 → 响应返回 9.90

图片路径约定：
  - 接口返回相对路径，前端拼接 CDN 域名
  - 示例："shop/2025/川味小馆_01.jpg"
        → "https://cdn.yuexiang.com/shop/2025/川味小馆_01.jpg"

时间格式约定：
  - 统一 ISO 8601：yyyy-MM-dd'T'HH:mm:ss
```

***

## 一、接口总览

```plain
本页共 6 个接口，围绕商户详情展开
进入页面并行加载前 4 个，后 2 个由用户操作 / 静默触发
```

| # | Method | URI                          | 对应区块         |
| - | ------ | ---------------------------- | ------------ |
| 1 | `GET`  | `/api/shop/{id}`             | 商户基本信息 + 轮播图 |
| 2 | `GET`  | `/api/shop/{id}/ai-analysis` | AI 口碑分析      |
| 3 | `GET`  | `/api/voucher/list`          | 该商户优惠券列表     |
| 4 | `GET`  | `/api/blog/of/shop`          | 相关探店笔记       |
| 5 | `PUT`  | `/api/favorite/toggle`       | 收藏 / 取消收藏    |
| 6 | `POST` | `/api/browse-history`        | 上报浏览记录       |

## 二、前置 DDL 变更

> 以下字段在原始建表脚本中缺失，需补充后方可支撑本页接口。

```sql
-- 1. tb_shop 增加联系电话字段
ALTER TABLE tb_shop
  ADD COLUMN phone VARCHAR(20) DEFAULT NULL
  COMMENT '联系电话' AFTER open_hours;

-- 2. tb_shop 增加收藏数冗余字段
ALTER TABLE tb_shop
  ADD COLUMN favorite_count INT UNSIGNED NOT NULL DEFAULT 0
  COMMENT '收藏数(异步同步)' AFTER review_count;

-- 3. AI 口碑分析持久化表（缓存兜底 + 历史趋势对比）
CREATE TABLE IF NOT EXISTS `tb_shop_ai_analysis` (
  `id`                BIGINT        NOT NULL AUTO_INCREMENT          COMMENT '主键ID',
  `shop_id`           BIGINT        NOT NULL                         COMMENT '商户ID',
  `overall_score`     DECIMAL(2,1)  NOT NULL                         COMMENT '综合评分',
  `score_level`       VARCHAR(16)   NOT NULL                         COMMENT '评分等级(excellent/good/average/poor)',
  `dimensions`        JSON          DEFAULT NULL                     COMMENT '多维评分',
  `positive_keywords` JSON          DEFAULT NULL                     COMMENT '好评关键词',
  `negative_keywords` JSON          DEFAULT NULL                     COMMENT '差评关键词',
  `summary`           VARCHAR(1024) DEFAULT NULL                     COMMENT 'AI综合摘要',
  `based_on_count`    INT           NOT NULL DEFAULT 0               COMMENT '分析依据条数',
  `trending`          VARCHAR(8)    NOT NULL DEFAULT 'stable'        COMMENT '趋势(up/stable/down)',
  `insufficient`      TINYINT       NOT NULL DEFAULT 0               COMMENT '数据是否不足(0否,1是)',
  `create_time`       DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `update_time`       DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP
                                    ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_shop_analysis` (`shop_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='商户AI口碑分析表';
```

## 四、核心实现逻辑

### 4.1 页面加载时序

```plain
用户点击商户卡片 → 进入详情页
│
├── 并行请求（不阻塞渲染）
│   ├── ① GET /api/shop/{id}              ← 最优先，骨架屏依赖
│   ├── ② GET /api/voucher/list?shopId=    ← 券区块
│   ├── ③ GET /api/blog/of/shop?shopId=    ← 笔记区块
│   └── ④ GET /api/shop/{id}/ai-analysis   ← AI 分析（可懒加载）
│
├── 静默上报（fire & forget，失败不重试）
│   └── ⑤ POST /api/browse-history
│
└── 用户操作触发
    └── ⑥ PUT /api/favorite/toggle         ← 点击收藏按钮
```

### 4.2 商户详情查询

```sql
-- 主查询
SELECT s.*,
       st.name AS type_name
  FROM tb_shop s
  JOIN tb_shop_type st ON st.id = s.type_id
 WHERE s.id = #{id}
   AND s.deleted = 0;
```

### 4.3 AI 口碑分析生成

```plain
生成策略（离线预计算 + 持久化 + 缓存，非实时调用 LLM）
│
├── 数据采集
│   ├── tb_review  → 评分数据 + 文字评价（status=1 已发布）
│   └── tb_blog    → 探店笔记正文（status=1 已发布）
│
├── 多维评分计算
│   SELECT
│     ROUND(AVG(score_taste),   1) AS avg_taste,
│     ROUND(AVG(score_env),     1) AS avg_env,
│     ROUND(AVG(score_service), 1) AS avg_service
│   FROM tb_review
│   WHERE shop_id = #{shopId}
│     AND status  = 1
│     AND deleted = 0;
│
├── 关键词提取（定时任务 / 评价新增时触发）
│   ├── 1. 合并该商户所有评价文本 + 笔记文本
│   ├── 2. 调用 NLP 服务 / LLM 进行情感分析
│   │   ├── 正面关键词 + 出现频次
│   │   └── 负面关键词 + 出现频次
│   ├── 3. 生成综合摘要（summary）
│   └── 4. 结果写入 tb_shop_ai_analysis 持久化
│         + 写入缓存 shop:ai:analysis:{shopId}
│
├── 口碑趋势判断
│   ├── 近 30 天平均分 vs 历史平均分
│   ├── 差值 > +0.1  → trending = "up"
│   ├── 差值 < -0.1  → trending = "down"
│   └── |差值| ≤ 0.1 → trending = "stable"
│
├── 数据不足处理
│   ├── basedOnCount < 10 → insufficient = true
│   ├── dimensions = null
│   ├── positiveKeywords = []，negativeKeywords = []
│   └── summary = "该商户评价较少，暂无法生成详细分析"
│
├── 读取链路
│   ├── 优先读缓存 shop:ai:analysis:{shopId}
│   ├── 未命中读 tb_shop_ai_analysis 并回写缓存
│   └── 均未命中 → 触发异步生成任务 + 返回 insufficient=true
│
└── 触发更新条件
    ├── 新增评价时 → 异步消息触发重算
    ├── 新增笔记时 → 异步消息触发重算
    └── 每日 03:00 定时任务 → 全量商户批量更新
```

### 4.5 收藏 Toggle（原子操作）

```plain
Service 流程（使用 INSERT ON DUPLICATE KEY UPDATE 避免竞态）：
│
├── 1. 原子切换
│     INSERT INTO tb_favorite (user_id, biz_type, biz_id, deleted, create_time)
│     VALUES (#{userId}, #{bizType}, #{bizId}, 0, NOW())
│     ON DUPLICATE KEY UPDATE
│       deleted = IF(deleted = 0, 1, 0),
│       create_time = IF(deleted = 1, NOW(), create_time);
│
│     -- 利用唯一键 uk_favorite_user_biz (user_id, biz_type, biz_id)
│     -- deleted=0 → 更新为 1（取消收藏）
│     -- deleted=1 → 更新为 0（重新收藏）
│     -- 无记录   → 插入 deleted=0（首次收藏）
│
├── 2. 判断操作结果
│     ├── ROW_COUNT() = 1 且为 INSERT → 收藏
│     ├── ROW_COUNT() = 2 → UPDATE 发生
│     │   └── 再查一次 deleted 确定最终状态
│     └── 据此决定 INCR 还是 DECR
│
├── 3. 更新收藏计数
│     ├── 收藏 → Redis INCR shop:favorite:count:{shopId}
│     └── 取消 → Redis DECR shop:favorite:count:{shopId}
│     └── 定时任务每 10 分钟将 Redis 计数回写 tb_shop.favorite_count
│
├── 4. 前端防抖
│     前端按钮点击后 300ms 内不可重复触发
│
└── 5. 返回最新状态 + 计数
```

### 4.7 浏览记录上报（含防刷）

```plain
Service 流程：
│
├── 1. 短时间去重（5 分钟窗口）
│     String dedupeKey = "browse:dedup:" + userId + ":" + bizType + ":" + bizId;
```

### 缓存穿透防护

```plain
商户详情热点接口防护策略：
│
├── 空值缓存
│   shop:detail:{id} = "" （TTL 2min）
│   → 不存在的 ID 查库一次后缓存空值
│   → 第二次请求直接返回 404，不穿透
│
├── 布隆过滤器（可选，商户量 > 10w 时建议启用）
│   bloom:shop:ids → 全量商户 ID
│   → 请求到达先判断 ID 是否存在
│   → 不存在则直接返回 404
│
└── 互斥锁重建
    缓存失效 → SETNX lock:shop:{id} EX 10
    → 获取锁的线程查库回写缓存
    → 未获取锁的线程休眠 50ms 后重试读缓存
    → 最多重试 3 次
```
