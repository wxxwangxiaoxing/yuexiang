## 一、技术架构设计

### 1.1 整体架构


### 1.2 技术选型

| 组件       | 技术方案             | 说明                                                     |
| ---------- | -------------------- | -------------------------------------------------------- |
| 消息队列   | RocketMQ             | 成熟MQ，自带重试、死信、消费组、顺序消费等能力            |
| 原子操作   | Redis Lua            | **仅操作 Redis**，保证点赞状态+计数原子性                 |
| 数据库事务 | Spring 事务          | 点赞记录 + 笔记点赞数在同一事务中写入                     |
| 消息发送   | 事务后发送           | DB事务提交后发送MQ消息，失败由校对任务兜底                 |
| 批处理     | 消费者本地聚合       | 5秒窗口批量聚合，减少DB写入次数                           |
| 失败处理   | RocketMQ重试+死信    | 自动重试16次，最终进入死信队列                            |
| 限流       | Redisson RateLimiter | Controller层用户维度限流                                  |

### 1.3 核心设计原则

```
┌──────────────────────────────────────────────────────────────┐
│                       设计原则                                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  1. Redis 与 MySQL 操作边界清晰                               │
│     • Lua 脚本只操作 Redis（状态、计数）                       │
│     • DB 操作在 Java 事务中完成                               │
│     • MQ 消息在 DB 事务提交后发送                              │
│     • DB 失败时回滚 Redis 状态，不发送 MQ                      │
│                                                              │
│  2. 消息可靠投递                                              │
│     • DB 事务提交成功后才发送 MQ 消息                          │
│     • MQ 发送失败不影响点赞结果（校对兜底）                     │
│     • RocketMQ 自带重试 + 死信机制                             │
│                                                              │
│  3. 数据最终一致                                              │
│     • 笔记点赞数：同步更新（强一致）                           │
│     • 用户获赞数：异步更新（最终一致，延迟 ≤ 5秒）              │
│     • 每日凌晨全量校对兜底                                    │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 二、数据流程设计

### 2.1 点赞完整流程

```
用户发起点赞请求
│
├─ ① Controller层限流
│   └─ @RateLimiter(key="like:{userId}", rate=1/s)
│
├─ ② Redis Lua 原子操作（仅操作Redis）
│   ├─ SISMEMBER blog:liked:{blogId} {userId}  → 检查是否已点赞
│   ├─ SADD blog:liked:{blogId} {userId}       → 添加点赞记录
│   └─ INCR blog:like:count:{blogId}           → 笔记点赞数 +1
│   └─ 返回: 1=成功, 0=已点赞
│
├─ ③ 数据库事务写入
│   ├─ INSERT tb_blog_likes (user_id, blog_id)
│   └─ UPDATE tb_blog SET like_count = like_count + 1
│   └─ 失败时 → 执行 Redis 回滚 Lua 脚本，不发送 MQ
│
├─ ④ 事务提交成功后发送 MQ 消息
│   └─ Topic: LIKE_EVENT_TOPIC
│   └─ Tag: LIKE
│   └─ Key: {blogId}:{userId}
│   └─ Body: LikeEvent(authorId, blogId, userId, delta=+1)
│   └─ 发送失败仅记录日志，不影响点赞结果（校对兜底）
│
└─ ⑤ 返回点赞成功
```

### 2.2 取消点赞完整流程

```
用户发起取消点赞请求
│
├─ ① Controller层限流
│
├─ ② Redis Lua 原子操作（仅操作Redis）
│   ├─ SISMEMBER blog:liked:{blogId} {userId}  → 检查是否已点赞
│   ├─ SREM blog:liked:{blogId} {userId}       → 移除点赞记录
│   └─ DECR blog:like:count:{blogId}           → 笔记点赞数 -1
│   └─ 返回: 1=成功, 0=未点赞
│
├─ ③ 数据库事务写入
│   ├─ DELETE FROM tb_blog_likes WHERE user_id=? AND blog_id=?
│   └─ UPDATE tb_blog SET like_count = GREATEST(like_count - 1, 0)
│   └─ 失败时 → 执行 Redis 回滚 Lua 脚本，不发送 MQ
│
├─ ④ 事务提交成功后发送 MQ 消息
│   └─ Topic: LIKE_EVENT_TOPIC
│   └─ Tag: UNLIKE
│   └─ Body: LikeEvent(authorId, blogId, userId, delta=-1)
│
└─ ⑤ 返回取消成功
```

### 2.3 异步消费流程

```
RocketMQ 消费者
│
├─ ① 接收消息（并发消费模式）
│   └─ 每次最多拉取 32 条
│
├─ ② 本地缓冲聚合
│   ├─ deltaBuffer: Map<authorId, delta> 增量合并
│   └─ 消费成功返回 CONSUME_SUCCESS（消息ACK）
│
├─ ③ 定时 flush（每5秒）
│   ├─ 快照并清空缓冲区
│   ├─ 批量更新 tb_user_info.like_count
│   └─ DB 失败 → 数据写回缓冲区，下次重试
│
└─ ④ 消费失败处理
    ├─ 返回 RECONSUME_LATER → RocketMQ 自动重试
    └─ 超过 16 次重试 → 自动进入 %DLQ% 死信队列
```

### 2.4 消息格式

```json
{
  "authorId": "笔记作者用户ID",
  "blogId": "笔记ID",
  "likeUserId": "点赞用户ID",
  "delta": "+1 或 -1",
  "type": "LIKE 或 UNLIKE",
  "timestamp": "毫秒时间戳"
}
```

### 3.3 SQL 语句

```sql
-- ① 笔记点赞数（同步事务中更新）
UPDATE tb_blog 
SET like_count = like_count + 1, update_time = NOW()
WHERE id = #{blogId} AND deleted = 0;

-- ② 笔记取消点赞数（下限保护）
UPDATE tb_blog 
SET like_count = GREATEST(like_count - 1, 0), update_time = NOW()
WHERE id = #{blogId} AND deleted = 0;

-- ③ 用户获赞数单条更新（下限保护）
UPDATE tb_user_info 
SET like_count = GREATEST(like_count + #{delta}, 0), update_time = NOW()
WHERE user_id = #{userId} AND deleted = 0;

-- ④ 用户获赞数批量更新（下限保护）
<foreach collection="entries" item="entry" separator=";">
    UPDATE tb_user_info 
    SET like_count = GREATEST(like_count + #{entry.delta}, 0), update_time = NOW()
    WHERE user_id = #{entry.userId} AND deleted = 0
</foreach>

-- ⑤ 全量校对：重新统计真实获赞数
UPDATE tb_user_info ui
SET ui.like_count = (
    SELECT COUNT(*) 
    FROM tb_blog_likes bl 
    INNER JOIN tb_blog b ON bl.blog_id = b.id 
    WHERE b.user_id = ui.user_id AND b.deleted = 0 AND bl.deleted = 0
), ui.update_time = NOW()
WHERE ui.user_id = #{userId} AND ui.deleted = 0;
```

## 四、Redis 设计

### 4.1 Key 设计

```
┌─────────────────────────────────────────────────────────────────────┐
│ Key                         │ 类型   │ TTL      │ 说明             │
├─────────────────────────────┼────────┼──────────┼─────────────────┤
│ blog:liked:{blogId}         │ SET    │ 永久     │ 点赞用户ID集合   │
│ blog:like:count:{blogId}    │ STRING │ 永久     │ 笔记点赞计数     │
│ user:like:count:{userId}    │ STRING │ 1小时    │ 用户获赞数缓存   │
└─────────────────────────────────────────────────────────────────────┘

说明：
• 使用 SET 代替独立 Key 存储点赞关系，大幅减少 Key 数量
  - 旧方案: blog:liked:{blogId}:{userId} → 5000万个Key（100万笔记×50点赞）
  - 新方案: blog:liked:{blogId}           → 100万个Key，内存减少约60%
  
• 相比 Stream 方案，Redis 不再承担消息队列职责，Key 更少更简洁
```

### 4.2 Lua 脚本 — 点赞

```lua
-- like.lua
-- KEYS[1] = blog:liked:{blogId}
-- KEYS[2] = blog:like:count:{blogId}
-- ARGV[1] = userId

-- 1. 检查是否已点赞
if redis.call('SISMEMBER', KEYS[1], ARGV[1]) == 1 then
    return 0  -- 已点赞
end

-- 2. 添加点赞记录
redis.call('SADD', KEYS[1], ARGV[1])

-- 3. 笔记点赞数+1
redis.call('INCR', KEYS[2])

return 1  -- 成功
```

### 4.3 Lua 脚本 — 取消点赞

```lua
-- unlike.lua
-- KEYS[1] = blog:liked:{blogId}
-- KEYS[2] = blog:like:count:{blogId}
-- ARGV[1] = userId

-- 1. 检查是否已点赞
if redis.call('SISMEMBER', KEYS[1], ARGV[1]) == 0 then
    return 0  -- 未点赞
end

-- 2. 移除点赞记录
redis.call('SREM', KEYS[1], ARGV[1])

-- 3. 笔记点赞数-1（Lua层无需下限保护，DB层兜底）
redis.call('DECR', KEYS[2])

return 1  -- 成功
```

### 4.4 Lua 脚本 — 回滚

```lua
-- rollback_like.lua
-- 当DB写入失败时回滚Redis状态
-- KEYS[1] = blog:liked:{blogId}
-- KEYS[2] = blog:like:count:{blogId}
-- ARGV[1] = userId
-- ARGV[2] = "LIKE" 或 "UNLIKE"

if ARGV[2] == 'LIKE' then
    -- 回滚点赞：移除记录，计数-1
    redis.call('SREM', KEYS[1], ARGV[1])
    redis.call('DECR', KEYS[2])
else
    -- 回滚取消点赞：恢复记录，计数+1
    redis.call('SADD', KEYS[1], ARGV[1])
    redis.call('INCR', KEYS[2])
end

return 1
```

## 五、RocketMQ 设计

### 5.1 Topic 与 Consumer Group 设计

```
┌───────────────────────────────────────────────────────────────────────┐
│                     RocketMQ 配置设计                                  │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  Topic: LIKE_EVENT_TOPIC                                              │
│  ├─ Tag: LIKE      (点赞事件)                                         │
│  ├─ Tag: UNLIKE    (取消点赞事件)                                     │
│  ├─ 读写队列数: 4  (根据消费者实例数调整)                               │
│  └─ 消息保留: 48小时                                                  │
│                                                                       │
│  Consumer Group: CG_LIKE_USER_COUNT                                   │
│  ├─ 消费模式: CLUSTERING (集群模式，每条消息只消费一次)                  │
│  ├─ 消费方式: PUSH (推模式)                                           │
│  ├─ 最大重试: 16次 (RocketMQ 默认)                                    │
│  ├─ 批量消费: maxConsumeSize = 32                                     │
│  └─ 消费线程: min=4, max=8                                           │
│                                                                       │
│  死信 Topic: %DLQ%CG_LIKE_USER_COUNT                                  │
│  └─ 超过16次重试后自动进入                                             │
│                                                                       │
│  死信 Consumer Group: CG_LIKE_DEAD_LETTER                             │
│  └─ 消费死信消息，持久化到 tb_like_dead_letter 表                      │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

### 5.2 消息发送策略

```
┌───────────────────────────────────────────────────────────────────────┐
│                     消息发送策略                                       │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  发送时机：DB 事务提交成功后                                           │
│                                                                       │
│  ┌─────────────────────────────────────────────────────────────┐      │
│  │  ① Lua 原子操作 Redis                                      │      │
│  │  ② DB 事务：INSERT 点赞记录 + UPDATE 笔记计数               │      │
│  │  ③ 事务提交成功                                             │      │
│  │  ④ 发送 MQ 消息（异步发送，SendCallback）                   │      │
│  │     ├─ 成功 → 正常流程                                     │      │
│  │     └─ 失败 → 记录日志，不影响用户操作                      │      │
│  │              （由每日校对任务兜底修复数据偏差）               │      │
│  └─────────────────────────────────────────────────────────────┘      │
│                                                                       │
│  为什么不用事务消息？                                                  │
│  • 点赞场景可容忍短暂不一致，事务消息复杂度高                           │
│  • MQ 发送失败概率极低（<0.01%），校对可完全修复                       │
│  • 事务消息需要实现回查接口，增加额外开发维护成本                       │
│                                                                       │
│  Key 设计：{blogId}:{userId}                                          │
│  • 用于消息去重和问题排查                                              │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

### 6.3 点赞事件

```java
@Data
@Builder
@NoArgsConstructor
```

### 6.5 点赞服务实现

```java
@Slf4j
@Service
@RequiredArgsConstructor
public class BlogLikeServiceImpl implements BlogLikeService {

    private final StringRedisTemplate redisTemplate;
    private final BlogMapper blogMapper;
    private final BlogLikeMapper blogLikeMapper;
    private final LikeEventProducer likeEventProducer;

    // 加载Lua脚本
    private static final DefaultRedisScript<Long> LIKE_SCRIPT;
    private static final DefaultRedisScript<Long> UNLIKE_SCRIPT;
    private static final DefaultRedisScript<Long> ROLLBACK_LIKE_SCRIPT;

    static {
        LIKE_SCRIPT = new DefaultRedisScript<>();
        LIKE_SCRIPT.setLocation(new ClassPathResource("lua/like.lua"));
        LIKE_SCRIPT.setResultType(Long.class);

        UNLIKE_SCRIPT = new DefaultRedisScript<>();
        UNLIKE_SCRIPT.setLocation(new ClassPathResource("lua/unlike.lua"));
        UNLIKE_SCRIPT.setResultType(Long.class);

        ROLLBACK_LIKE_SCRIPT = new DefaultRedisScript<>();
        ROLLBACK_LIKE_SCRIPT.setLocation(new ClassPathResource("lua/rollback_like.lua"));
        ROLLBACK_LIKE_SCRIPT.setResultType(Long.class);
    }

    @Override
    public void likeBlog(Long userId, Long blogId) {
        // 1. 查询笔记作者
        Blog blog = blogMapper.selectById(blogId);
        if (blog == null) {
            throw new BusinessException("笔记不存在");
        }
        Long authorId = blog.getUserId();

        // 2. Redis Lua 原子操作（仅操作Redis，不发MQ）
        List<String> keys = List.of(
                BLOG_LIKED_KEY + blogId,
                BLOG_LIKE_COUNT_KEY + blogId
        );
        Long result = redisTemplate.execute(LIKE_SCRIPT, keys, String.valueOf(userId));

        if (result == null || result == 0L) {
            throw new BusinessException("已点赞，请勿重复操作");
        }

        // 3. 数据库事务写入
        try {
            persistLike(userId, blogId);
        } catch (Exception e) {
            // DB失败 → 回滚Redis状态，不发送MQ
            rollbackRedis(userId, blogId, "LIKE");
            log.error("[Like] DB写入失败，已回滚Redis. userId={}, blogId={}", userId, blogId, e);
            throw new BusinessException("点赞失败，请重试");
        }

        // 4. DB事务成功后发送MQ消息（异步发送，失败不影响业务）
        LikeEvent event = LikeEvent.builder()
                .authorId(authorId)
                .blogId(blogId)
                .likeUserId(userId)
                .delta(1)
                .type(TAG_LIKE)
                .timestamp(System.currentTimeMillis())
                .build();
        likeEventProducer.sendLikeEvent(event);

        // 5. 清除用户获赞数缓存
        redisTemplate.delete(USER_LIKE_COUNT_KEY + authorId);

        log.info("[Like] 点赞成功. userId={}, blogId={}, authorId={}", userId, blogId, authorId);
    }

    @Override
    public void unlikeBlog(Long userId, Long blogId) {
        // 1. 查询笔记作者
        Blog blog = blogMapper.selectById(blogId);
        if (blog == null) {
            throw new BusinessException("笔记不存在");
        }
        Long authorId = blog.getUserId();

        // 2. Redis Lua 原子操作
        List<String> keys = List.of(
                BLOG_LIKED_KEY + blogId,
                BLOG_LIKE_COUNT_KEY + blogId
        );
        Long result = redisTemplate.execute(UNLIKE_SCRIPT, keys, String.valueOf(userId));

        if (result == null || result == 0L) {
            throw new BusinessException("未点赞，无法取消");
        }

        // 3. 数据库事务写入
        try {
            persistUnlike(userId, blogId);
        } catch (Exception e) {
            // DB失败 → 回滚Redis状态，不发送MQ
            rollbackRedis(userId, blogId, "UNLIKE");
            log.error("[Unlike] DB写入失败，已回滚Redis. userId={}, blogId={}", userId, blogId, e);
            throw new BusinessException("取消点赞失败，请重试");
        }

        // 4. DB事务成功后发送MQ消息
        LikeEvent event = LikeEvent.builder()
                .authorId(authorId)
                .blogId(blogId)
                .likeUserId(userId)
                .delta(-1)
                .type(TAG_UNLIKE)
                .timestamp(System.currentTimeMillis())
                .build();
        likeEventProducer.sendLikeEvent(event);

        // 5. 清除用户获赞数缓存
        redisTemplate.delete(USER_LIKE_COUNT_KEY + authorId);

        log.info("[Unlike] 取消点赞成功. userId={}, blogId={}, authorId={}", userId, blogId, authorId);
    }

    @Transactional(rollbackFor = Exception.class)
    protected void persistLike(Long userId, Long blogId) {
        blogLikeMapper.insert(userId, blogId);
        blogMapper.incrementLikeCount(blogId);
    }

    @Transactional(rollbackFor = Exception.class)
    protected void persistUnlike(Long userId, Long blogId) {
        blogLikeMapper.deleteByUserAndBlog(userId, blogId);
        blogMapper.decrementLikeCount(blogId);
    }

    private void rollbackRedis(Long userId, Long blogId, String type) {
        try {
            List<String> keys = List.of(
                    BLOG_LIKED_KEY + blogId,
                    BLOG_LIKE_COUNT_KEY + blogId
            );
            redisTemplate.execute(ROLLBACK_LIKE_SCRIPT, keys,
                    String.valueOf(userId), type);
        } catch (Exception rollbackEx) {
            log.error("[Like] Redis回滚也失败! userId={}, blogId={}, type={}",
                    userId, blogId, type, rollbackEx);
            // 告警，后续由每日校对任务修复
        }
    }
}
```

### 6.6 MQ 消费者 — 点赞事件

```java
@Slf4j
@Component
@RocketMQMessageListener(
        topic = LikeMQConstants.LIKE_EVENT_TOPIC,
        consumerGroup = LikeMQConstants.CG_LIKE_USER_COUNT,
        selectorExpression = "LIKE || UNLIKE",
        consumeMode = ConsumeMode.CONCURRENTLY,
        maxReconsumeTimes = 16
)
@RequiredArgsConstructor
public class LikeEventConsumer implements RocketMQListener<LikeEvent> {

    private final UserInfoMapper userInfoMapper;

    /** 本地增量缓冲区: authorId → delta累计值 */
    private final ConcurrentHashMap<Long, AtomicInteger> deltaBuffer = new ConcurrentHashMap<>();

    @Override
    public void onMessage(LikeEvent event) {
        try {
            // 消息聚合到本地缓冲区
            deltaBuffer.computeIfAbsent(event.getAuthorId(), k -> new AtomicInteger(0))
                    .addAndGet(event.getDelta());

            log.debug("[LikeConsumer] 消息聚合. authorId={}, delta={}, bufferSize={}",
                    event.getAuthorId(), event.getDelta(), deltaBuffer.size());

        } catch (Exception e) {
            log.error("[LikeConsumer] 消息处理异常. event={}", event, e);
            // 抛出异常让 RocketMQ 重试
            throw new RuntimeException("消费失败，等待重试", e);
        }
    }

    /**
     * 定时刷新缓冲区到数据库（每5秒执行一次）
     */
    @Scheduled(fixedRate = 5000)
    public void flushDeltaBuffer() {
        if (deltaBuffer.isEmpty()) {
            return;
        }

        // 1. 快照并清空缓冲区
        Map<Long, Integer> snapshot = new HashMap<>();
        deltaBuffer.forEach((userId, delta) -> {
            int value = delta.getAndSet(0);
            if (value != 0) {
                snapshot.put(userId, value);
            }
        });
        // 清理 delta=0 的空条目
        deltaBuffer.entrySet().removeIf(e -> e.getValue().get() == 0);

        if (snapshot.isEmpty()) {
            return;
        }

        // 2. 批量更新数据库
        try {
            userInfoMapper.batchUpdateLikeCount(snapshot);
            log.info("[LikeConsumer] 批量更新用户获赞数. userCount={}, details={}",
                    snapshot.size(), snapshot);

        } catch (Exception e) {
            log.error("[LikeConsumer] 批量更新失败，数据写回缓冲区. snapshot={}", snapshot, e);
            // 写回缓冲区，下次 flush 重试
            snapshot.forEach((userId, delta) ->
                    deltaBuffer.computeIfAbsent(userId, k -> new AtomicInteger(0))
                            .addAndGet(delta));
        }
    }
}
```

### 6.8 每日全量校对任务

```java
@Slf4j
@Component
@RequiredArgsConstructor
public class LikeCountReconcileTask {

    private final UserInfoMapper userInfoMapper;
    private final BlogLikeMapper blogLikeMapper;

    private static final int BATCH_SIZE = 500;

    /**
     * 每日凌晨3点全量校对用户获赞数
     * 以数据库点赞记录为准，修正 tb_user_info.like_count
     */
    @Scheduled(cron = "0 0 3 * * ?")
    public void reconcile() {
        log.info("[Reconcile] 开始每日获赞数校对");
        long startTime = System.currentTimeMillis();
        int totalUsers = 0;
        int fixedUsers = 0;

        int page = 0;
        while (true) {
            // 分页查询用户
            List<Long> userIds = userInfoMapper.selectUserIdsByPage(page * BATCH_SIZE, BATCH_SIZE);
            if (userIds.isEmpty()) break;

            for (Long userId : userIds) {
                totalUsers++;
                try {
                    // 统计真实获赞数
                    long realCount = blogLikeMapper.countByAuthorId(userId);
                    // 查询当前记录值
                    Long currentCount = userInfoMapper.selectLikeCount(userId);

                    if (currentCount == null || currentCount != realCount) {
                        userInfoMapper.updateLikeCountDirectly(userId, realCount);
                        fixedUsers++;
                        log.info("[Reconcile] 修正用户获赞数: userId={}, {} → {}",
                                userId, currentCount, realCount);
                    }
                } catch (Exception e) {
                    log.error("[Reconcile] 校对失败: userId={}", userId, e);
                }
            }

            page++;
        }

        long elapsed = System.currentTimeMillis() - startTime;
        log.info("[Reconcile] 校对完成. 总用户={}, 修正={}, 耗时={}ms",
                totalUsers, fixedUsers, elapsed);
    }
}
```

### 8.1 三级一致性保障

```
┌──────────────────────────────────────────────────────────────┐
│                   数据一致性保障体系                           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  第一级：实时保障                                             │
│  ├─ Lua 脚本保证 Redis 操作原子性                             │

│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 8.2 幂等性保证

```
点赞幂等：
  • Lua 脚本 SISMEMBER 检查，重复点赞返回 0，业务层抛异常
  • DB 层 UNIQUE KEY (user_id, blog_id) 兜底

消费幂等：
  • 增量聚合模式天然幂等（+1 -1 抵消）
  • 即使重复消费，每日校对会修正

全量校对幂等：
  • 直接 SET 而非增量更新，多次执行结果相同
```

## 十三、故障演练清单

| 故障场景             | 预期表现                                    | 恢复方式             |
| -------------------- | ------------------------------------------ | -------------------- |
| 点赞时 DB 写入失败   | Redis 自动回滚，不发送MQ，用户可重试         | 自动恢复             |
| Redis 回滚也失败     | 日志告警，数据暂不一致                       | 每日校对修复         |
| DB 成功 MQ 发送失败  | 点赞成功返回，MQ失败仅记录日志               | 每日校对修复         |
| MQ Broker 不可用     | 点赞/取消正常（同步部分），用户获赞数延迟更新  | MQ恢复后自动消费     |
| 消费者应用崩溃       | 消息保留在MQ中                              | 重启后自动重新消费   |
| flush 写 DB 失败     | 数据写回缓冲区，下次重试                     | 自动重试             |
| 消息消费失败超16次   | 自动进入死信队列，死信消费者持久化到DB         | 人工排查处理         |
| RocketMQ 主从切换    | 短暂不可用后自动恢复                         | 自动恢复             |
| 全量校对执行慢       | 分页处理，不影响业务                         | 调整 batch_size      |
